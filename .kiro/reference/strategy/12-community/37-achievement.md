# 37. 成就/勋章系统

## 是什么
用户完成特定行为后获得成就/勋章/积分，游戏化激励持续使用。通过收集欲、进度感、社交展示来提升用户留存和活跃度，让"使用产品"变成"玩产品"。

## 为什么需要
- 提升留存：收集欲驱动用户持续使用（"还差 2 个勋章集齐"）
- 引导探索：通过成就引导用户发现未使用的功能
- 增加趣味性：骑行本身就有游戏化潜力（里程/速度/时长）
- 社交展示：勋章墙是用户身份的一部分
- 数据洞察：成就完成率反映功能使用深度

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              成就/勋章系统架构                        │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [行为事件]                                          │
│   用户操作 → 事件总线                               │
│   ├─ 骑行事件：里程/时长/速度/频次                 │
│   ├─ 功能事件：使用灯效/音效/OTA/分享             │
│   ├─ 社交事件：发帖/回复/点赞/邀请               │
│   └─ 时间事件：连续使用天数/注册周年              │
│       │                                              │
│       ▼                                              │
│  [成就引擎]                                          │
│   事件 → 条件判断 → 进度更新 → 达成检测           │
│       │                                              │
│       ├─ [条件评估器]                               │
│       │   累计型：总里程 > 100km                    │
│       │   连续型：连续 7 天使用                     │
│       │   瞬时型：单次速度 > 40km/h                │
│       │   组合型：使用全部 10 种灯效               │
│       │                                              │
│       ▼                                              │
│  [发放与展示]                                        │
│   解锁动画 → 推送通知 → 勋章墙展示               │
│   积分发放 → 等级提升 → 特权解锁                  │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 事件系统 | 事件总线 (Redis Pub/Sub) | 解耦行为和成就判定 |
| 成就定义 | JSON 配置 + 数据库 | 灵活新增不发版 |
| 进度存储 | PostgreSQL JSONB | 每用户每成就的进度 |
| 通知 | App 推送 + 应用内弹窗 | 解锁时即时反馈 |
| 展示 | SVG/PNG 勋章图标 | 精美视觉设计 |
| 排行榜 | Redis Sorted Set | 实时排名 |

## 实现步骤

### Phase 1：成就定义框架（2h）
1. 成就数据模型：
   ```sql
   CREATE TABLE achievements (
     id          VARCHAR(50) PRIMARY KEY,
     category    VARCHAR(30),    -- mileage/explore/social/collect/time
     name        VARCHAR(100),
     description VARCHAR(200),
     icon_url    VARCHAR(500),
     rarity      VARCHAR(20),    -- common/rare/epic/legendary
     condition   JSONB,          -- 达成条件定义
     reward      JSONB,          -- 奖励（积分/称号/特权）
     sort_order  INT,
     created_at  TIMESTAMPTZ DEFAULT NOW()
   );

   CREATE TABLE user_achievements (
     user_id       UUID REFERENCES users(id),
     achievement_id VARCHAR(50) REFERENCES achievements(id),
     progress      JSONB,        -- 当前进度
     unlocked      BOOLEAN DEFAULT false,
     unlocked_at   TIMESTAMPTZ,
     PRIMARY KEY (user_id, achievement_id)
   );
   ```
2. 成就分类设计：
   ```
   里程碑类：首次骑行/累计10km/100km/1000km
   探索类：使用全部灯效/尝试全部音效/完成OTA
   社交类：首次分享/邀请好友/发帖获赞
   收集类：集齐所有颜色/解锁全部模式
   时间类：连续7天/30天/注册周年
   ```
3. 稀有度设计：普通(60%) / 稀有(25%) / 史诗(10%) / 传说(5%)

### Phase 2：成就引擎实现（3h）
4. 条件评估器：
   ```javascript
   const conditionTypes = {
     cumulative: (progress, target) => progress.total >= target,
     streak: (progress, target) => progress.current_streak >= target,
     instant: (value, target) => value >= target,
     collection: (progress, required) => 
       required.every(item => progress.collected.includes(item)),
   };
   ```
5. 事件处理流程：
   ```
   用户骑行结束 → 发送 ride_complete 事件
   → 成就引擎接收
   → 更新相关成就进度（里程+距离，连续天数+1）
   → 检查是否达成
   → 达成 → 标记解锁 + 发放奖励 + 推送通知
   ```
6. 批量检查优化：按事件类型路由到相关成就，不全量扫描

### Phase 3：进度追踪（2h）
7. 进度数据结构：
   ```json
   {
     "achievement_id": "ride_100km",
     "progress": {
       "total_km": 67.5,
       "target_km": 100,
       "percentage": 67
     }
   }
   ```
8. 进度展示：进度条 + 百分比 + 剩余量提示
9. 里程碑提醒："还差 5km 就能解锁'百公里骑士'勋章！"

### Phase 4：展示与社交（2h）
10. 勋章墙：个人主页展示已解锁勋章（按稀有度排列）
11. 解锁动画：全屏动画 + 音效 + 分享按钮
12. 社交分享：生成精美分享卡片（勋章+数据+二维码）
13. 排行榜：勋章总数排行 / 里程排行 / 连续天数排行

### Phase 5：积分与兑换（1h）
14. 积分规则：
    - 解锁普通成就 +10 / 稀有 +30 / 史诗 +100 / 传说 +500
    - 每日签到 +5 / 连续签到加成
15. 积分用途：
    - 兑换高级灯效/音效包
    - 兑换社区特权（自定义头衔/彩色昵称）
    - 未来：兑换实物优惠券
16. 等级体系：积分累计 → 等级提升 → 解锁特权

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 成就太容易 | 没有成就感 | 阶梯设计，后期成就有挑战性 |
| 成就太难 | 用户放弃 | 大多数用户能完成 60% 的成就 |
| 进度丢失 | 用户愤怒 | 服务端存储 + 本地缓存双保险 |
| 作弊 | 刷数据解锁 | 服务端校验 + 异常检测 |
| 通知骚扰 | 频繁弹窗 | 合并通知 + 用户可关闭 |
| 设计膨胀 | 成就太多失去意义 | 控制总数 30-50 个，定期清理 |

## 与 RideWind 的关系
- 当前状态：无成就系统，无游戏化元素
- 优先级：P5（产品功能完善 + 有数据采集后）
- 可能的成就：

  - 使用天数（7/30/100/365天）
  - 功能探索（使用全部灯效/音效/模式）
  - 社交（分享/邀请/发帖）
- 依赖：#15 数据埋点（行为事件源）+ #2 用户系统
