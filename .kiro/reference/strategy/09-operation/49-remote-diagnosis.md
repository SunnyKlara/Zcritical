# 49. 远程诊断

## 是什么
远程查看设备运行状态、日志、错误码，不需要用户寄回设备就能定位问题。通过 App 代理或 MQTT 通道，从云端拉取设备实时/历史数据，结合错误码字典和诊断规则，快速定位故障原因并给出解决方案。

## 为什么需要
- 售后效率：不用寄回设备，远程 5 分钟定位问题 vs 来回快递 5 天
- 快速定位：设备日志 + 错误码 + 运行参数 = 精准定位
- 减少退货率：很多"故障"其实是使用问题，远程指导即可解决
- 数据积累：诊断数据反哺产品改进（#50 售后闭环）
- 用户体验：用户感受到"被重视"，提升品牌好感

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              远程诊断系统架构                         │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [设备端]                                            │
│   运行日志环形缓冲 → 错误码记录 → 健康指标采集    │
│       │                                              │
│       ▼                                              │
│  [传输通道]                                          │
│   ├─ BLE 通道：App 连接时实时拉取                   │
│   ├─ MQTT 通道：设备有网时主动上报                  │
│   └─ App 代理：BLE→App→Cloud 转发                  │
│       │                                              │
│       ▼                                              │
│  [诊断引擎]（云端）                                  │
│   日志解析 → 错误码匹配 → 规则引擎 → 诊断报告    │
│       │                                              │
│       ├─ [错误码字典] 0x0001=BLE初始化失败          │
│       ├─ [诊断规则] IF 温度>80℃ THEN 过热保护      │
│       └─ [解决方案库] 错误码→建议操作               │
│       │                                              │
│       ▼                                              │
│  [输出]                                              │
│   诊断报告 → 客服查看 / 用户自助 / 自动修复       │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 设备日志 | 环形缓冲区 (NVS/PSRAM) | 最近 N 条日志持久化 |
| 日志上报 | BLE 批量传输 / MQTT | 按需拉取或定时上报 |
| 错误码 | 枚举定义 + JSON 字典 | 设备端记录，云端解释 |
| 诊断引擎 | 规则引擎 (Node.js) | IF-THEN 规则匹配 |
| 远程命令 | BLE/MQTT 下发 | 重启/清缓存/恢复出厂 |
| 报告存储 | PostgreSQL + OSS | 结构化数据 + 日志文件 |
| 可视化 | Web 管理后台 | 客服查看诊断结果 |

## 实现步骤

### Phase 1：设备端日志系统（2-3h）
1. 环形日志缓冲区设计：
   ```c
   #define LOG_BUFFER_SIZE 100  // 最近 100 条日志
   
   typedef struct {
       uint32_t timestamp;     // 运行时间 ms
       uint8_t  level;         // ERROR/WARN/INFO
       uint16_t error_code;    // 错误码
       char     message[64];   // 描述
   } log_entry_t;
   
   typedef struct {
       log_entry_t entries[LOG_BUFFER_SIZE];
       uint16_t head;
       uint16_t count;
   } log_ring_buffer_t;
   ```
2. 错误码定义：
   ```c
   typedef enum {
       ERR_NONE           = 0x0000,
       ERR_BLE_INIT       = 0x0001,
       ERR_LCD_INIT       = 0x0002,
       ERR_AUDIO_INIT     = 0x0003,
       ERR_NVS_READ       = 0x0010,
       ERR_NVS_WRITE      = 0x0011,
       ERR_OTA_VERIFY     = 0x0020,
       ERR_OTA_WRITE      = 0x0021,
       ERR_TEMP_HIGH      = 0x0030,
       ERR_PSRAM_FAIL     = 0x0040,
   } error_code_t;
   ```
3. 健康指标采集：CPU 温度、运行时间、重启次数、BLE 连接次数、错误计数

### Phase 2：日志传输协议（2h）
4. BLE 诊断命令：
   ```
   CMD_DIAG_GET_LOGS:    → 返回最近 N 条日志
   CMD_DIAG_GET_STATUS:  → 返回当前健康指标
   CMD_DIAG_GET_ERRORS:  → 返回错误码历史
   CMD_DIAG_CLEAR_LOGS:  → 清除日志缓冲
   CMD_DIAG_RESTART:     → 远程重启设备
   CMD_DIAG_FACTORY:     → 恢复出厂设置
   ```
5. 数据压缩：日志批量传输时 LZ4 压缩，减少 BLE 传输时间
6. App 端代理上报：BLE 拉取日志 → HTTPS 上传云端

### Phase 3：云端诊断引擎（2-3h）
7. 错误码字典服务：
   ```json
   {
     "0x0001": {
       "name": "BLE 初始化失败",
       "severity": "critical",
       "possible_causes": ["蓝牙硬件故障", "NVS 数据损坏"],
       "solutions": ["恢复出厂设置", "重新烧录固件", "硬件返修"]
     }
   }
   ```
8. 诊断规则引擎：
   ```javascript
   const rules = [
     { condition: (d) => d.restart_count > 10 && d.uptime < 60,
       diagnosis: "频繁重启，可能固件崩溃",
       action: "建议回滚固件版本" },
     { condition: (d) => d.temp > 75,
       diagnosis: "设备过热",
       action: "检查散热/降低LED亮度" },
     { condition: (d) => d.ble_fail_count > 50,
       diagnosis: "BLE 连接不稳定",
       action: "检查天线/更新固件" }
   ];
   ```
9. 自动生成诊断报告：汇总设备状态 + 错误分析 + 建议方案

### Phase 4：客服集成（1-2h）
10. 客服工作台集成：工单关联设备 → 一键拉取诊断数据
11. 诊断报告展示：时间线视图（错误发生顺序）+ 健康仪表盘
12. 远程操作：客服可远程执行重启/清缓存/恢复出厂（需用户授权）

### Phase 5：用户自助诊断（1h）
13. App 内"设备体检"功能：一键检测 → 显示健康评分
14. 常见问题自动匹配：根据错误码推荐解决方案
15. 一键修复：自动执行安全的修复操作（清缓存、重连BLE）

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 日志占用内存 | 影响正常功能 | 环形缓冲 + 固定大小 + 溢出覆盖 |
| 隐私问题 | 用户不愿意上传数据 | 明确告知 + 用户授权 + 数据脱敏 |
| 错误码不全 | 新问题无法识别 | 未知错误码兜底 + 持续补充字典 |
| 远程操作风险 | 误操作导致设备异常 | 操作确认 + 审计日志 + 可回滚 |
| BLE 断连 | 日志传输中断 | 分批传输 + 断点续传 |
| 时间戳不准 | 日志时序混乱 | 用相对时间（运行时长）而非绝对时间 |
