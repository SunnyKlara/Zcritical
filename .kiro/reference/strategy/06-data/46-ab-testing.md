# 46. A/B 测试平台

## 是什么
同时运行多个版本的功能或 UI，将用户随机分流到不同组，通过数据对比选出最优方案。用统计学方法验证"改动是否真的有效"，而非靠直觉或老板拍板。

## 为什么需要
- 产品决策不应该靠"我觉得"，而是"数据证明"
- 降低试错成本：新方案只影响 10% 用户，不好就回滚
- 量化改进效果：新 UI 到底提升了多少转化率？
- 避免 HiPPO（Highest Paid Person's Opinion）决策
- 持续优化：每次小改进累积成大优势

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              A/B 测试平台架构                         │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [实验管理]                                          │
│   创建实验 → 定义假设 → 选择指标 → 设置分流      │
│       │                                              │
│       ▼                                              │
│  [分流引擎]                                          │
│   hash(user_id + experiment_id) % 100               │
│   ├─ 对照组 (Control): 50% → 旧版本              │
│   └─ 实验组 (Treatment): 50% → 新版本            │
│       │                                              │
│       ▼                                              │
│  [数据采集]                                          │
│   各组用户行为事件 → 按实验分组标记               │
│       │                                              │
│       ▼                                              │
│  [统计分析]                                          │
│   样本量检验 → 显著性检验 (p-value < 0.05)        │
│   置信区间 → 效果量 → 结论                        │
│       │                                              │
│       ▼                                              │
│  [决策]                                              │
│   显著优于对照 → 全量发布                          │
│   无显著差异 → 继续观察或放弃                      │
│   显著劣于对照 → 回滚                              │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 分流 | Feature Flag (#14) + hash | 复用已有 Flag 基础设施 |

## 实现步骤

### Phase 1：实验设计框架（2h）
1. 实验模板定义：
   ```json
   {
     "id": "exp_new_ota_ui",
     "hypothesis": "新 OTA 界面能提升升级完成率",
     "metric_primary": "ota_complete_rate",
     "metric_secondary": ["ota_start_rate", "time_to_complete"],
     "traffic_split": {"control": 50, "treatment": 50},
     "min_sample_size": 500,
     "duration_days": 14,
     "status": "running"
   }
   ```
2. 假设→指标→分流→观察→结论的标准流程
3. 实验互斥：同一用户不同时参与冲突实验

### Phase 2：分流引擎（2-3h）
4. 分流算法：
   ```python
   def get_variant(user_id, experiment_id, traffic_split):
       hash_val = hash(f"{user_id}:{experiment_id}") % 100
       if hash_val < traffic_split["control"]:
           return "control"
       else:
           return "treatment"
   ```
5. 特性：
   - 确定性：同一用户每次请求分到同一组
   - 均匀性：各组用户数接近预期比例
   - 独立性：不同实验的分流互不影响
6. 与 Feature Flag 集成：实验组看到新功能，对照组看到旧功能

### Phase 3：统计分析引擎（2-3h）
7. 样本量计算（实验前）：
   ```python
   from statsmodels.stats.power import TTestIndPower
   # 检测 5% 的提升，80% 统计功效，5% 显著性水平
   analysis = TTestIndPower()
   sample_size = analysis.solve_power(
       effect_size=0.05/baseline_std,
       power=0.8, alpha=0.05
   )
   ```
8. 显著性检验（实验中/后）：
   ```python
   from scipy import stats
   t_stat, p_value = stats.ttest_ind(control_data, treatment_data)
   if p_value < 0.05:
       print("统计显著，实验组优于对照组")
   ```
9. 置信区间计算：效果量的 95% 置信区间
10. 多重比较校正：多个指标时用 Bonferroni 校正

### Phase 4：实验管理界面（2h）
11. 实验列表：状态（草稿/运行中/已结束）、进度、结果
12. 创建实验：填写假设、选择指标、设置分流比例
13. 实时监控：各组样本量、核心指标趋势、预计结束时间
14. 结果报告：自动生成结论（显著/不显著/需要更多数据）

### Phase 5：安全机制（1h）
15. 提前终止规则：实验组指标严重恶化时自动停止
16. 渐进式放量：先 5% → 观察 → 20% → 50% → 全量
17. 回滚机制：一键将所有用户切回对照组

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 样本量不够 | 结论不可靠（假阳性） | 实验前计算最小样本量 |
| 偷看数据 | 过早下结论（peeking problem） | 预设实验周期，到期再看 |
| 新奇效应 | 新功能初期数据好，后期回落 | 观察足够长时间（2周+） |
| 辛普森悖论 | 分组不均匀导致误判 | 检查各组用户特征分布 |
| 多重比较 | 看 20 个指标总能找到"显著"的 | 预设主要指标 + Bonferroni 校正 |
| 实验污染 | 对照组用户看到实验组内容 | 严格隔离 + 检查泄漏 |
