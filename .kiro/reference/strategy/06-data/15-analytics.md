# 15. 数据埋点与分析

## 是什么
在 App 和设备中埋入数据采集点，记录用户行为（点击、页面浏览、功能使用）和设备状态（在线时长、错误率、性能指标），通过数据管线汇聚到分析平台，用于产品决策和体验优化。

## 为什么需要
- 不知道用户怎么用产品 = 盲目开发功能
- 哪些功能受欢迎？哪些从没人用？数据说了算
- 用户在哪一步流失？注册→绑定→使用的转化漏斗
- 设备异常率多少？哪个固件版本 crash 最多？
- 增长决策：投放渠道效果、留存率、用户生命周期价值

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              数据埋点架构                             │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [采集层]                                            │
│   App 端 SDK → 事件上报（点击/页面/自定义）        │
│   设备端 → 状态上报（通过 App 代理或 MQTT）        │
│       │                                              │
│       ▼                                              │
│  [传输层]                                            │
│   HTTP 批量上报 / Kafka 消息队列                    │
│   本地缓存 → 网络恢复后补传                        │
│       │                                              │
│       ▼                                              │
│  [存储层]                                            │
│   ClickHouse / BigQuery / Firebase                  │
│   原始事件表 → 聚合表 → 指标表                     │
│       │                                              │
│       ▼                                              │
│  [分析层]                                            │
│   看板 / 漏斗 / 留存 / 路径分析                    │
│   Grafana / Metabase / Firebase Console             │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| App 端 SDK | Firebase Analytics | 免费、跨平台、自动事件 |
| 备选 | 友盟 / GrowingIO | 国内合规、中文支持 |
| 自建采集 | Kafka + ClickHouse | 完全可控、无数据量限制 |
| 数据管线 | Kafka → Flink → ClickHouse | 实时 + 批处理 |
| 可视化 | Firebase Console / Metabase | 开箱即用 |
| 隐私合规 | GDPR/个保法 SDK 配置 | 用户同意后才采集 |

## 实现步骤

### Phase 1：事件设计规范（1-2h）
1. 定义事件命名规范：`{对象}_{动作}`（如 `device_connect`、`ota_start`）
2. 定义公共属性：user_id、device_sn、app_version、os、timestamp
3. 定义事件属性：每个事件的专属参数（如 ota_start 带 firmware_version）
4. 建立事件文档（Excel/Notion），开发前先评审

### Phase 2：SDK 集成（2-3h）
5. Flutter 集成 `firebase_analytics` 插件
6. 配置 Firebase 项目（iOS + Android）
7. 自动事件：页面浏览（screen_view）、App 启动（app_open）
8. 自定义事件封装：统一 `AnalyticsService.logEvent(name, params)`

### Phase 3：关键事件埋点（2-3h）
9. 核心漏斗：启动→注册→绑定设备→首次使用→7日留存
10. 功能使用：每个功能页面的进入/操作/退出
11. 设备事件：连接成功/失败、OTA 开始/完成/失败、错误上报
12. 性能事件：BLE 连接耗时、页面加载耗时

### Phase 4：数据验证与看板（1-2h）
13. Firebase DebugView 实时验证事件上报
14. 配置转化漏斗（注册→绑定→活跃）
15. 配置留存报告（D1/D7/D30）
16. 自定义看板：日活、功能使用排行、错误率

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 事件命名混乱 | 数据无法聚合分析 | 严格命名规范 + 代码审查 |
| 埋点遗漏 | 关键路径无数据 | 需求评审时同步确认埋点方案 |
| 数据量爆炸 | 存储成本高、查询慢 | 采样 + 聚合 + 保留策略 |
| 隐私合规 | 被罚款/下架 | 用户同意弹窗 + 匿名化 + 可删除 |
| 埋点影响性能 | App 卡顿 | 异步上报 + 批量发送 + 本地缓存 |
| 数据不准 | 决策依据错误 | 数据验证流程 + A/B 对照 |

## 与 RideWind 的关系
- 当前状态：App 无任何埋点，设备无状态上报
- 优先级：P4（App 上架后尽快加入）
- 起步方案：Firebase Analytics（免费、Flutter 原生支持、5 分钟集成）
- 关键指标：日活、设备连接成功率、功能使用分布、OTA 成功率
- 成本：Firebase 免费额度足够初期使用
