# 16. 数据看板/BI

## 是什么
将原始数据（用户行为、设备状态、业务指标）转化为可视化图表和报表，让团队成员无需写 SQL 就能实时了解业务健康度、发现问题和增长机会。是数据驱动决策的"眼睛"。

## 为什么需要
- 数据存在数据库里没人看 = 白采集
- 老板/运营不会写 SQL，需要图表化展示
- 实时了解业务健康度：今天日活多少？OTA 覆盖率多少？
- 发现异常：某天注册量突然下降 → 是 bug 还是渠道问题？
- 对齐目标：团队看同一个看板，对"好不好"有共识

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              BI 看板架构                              │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [数据源]                                            │
│   PostgreSQL (业务数据)                              │
│   ClickHouse (事件数据)                             │
│   Redis (实时指标)                                  │
│       │                                              │
│       ▼                                              │
│  [数据仓库层]（可选，规模大时）                     │
│   ETL 定时任务 → 聚合表 / 宽表                     │
│   dbt (数据转换) / Airflow (调度)                   │
│       │                                              │
│       ▼                                              │
│  [BI 工具]                                           │
│   Metabase / Grafana / Superset                     │
│   ├─ 战略看板：北极星指标、月度趋势               │
│   ├─ 战术看板：功能使用、转化漏斗                 │
│   └─ 操作看板：实时在线、错误告警                 │
│       │                                              │
│       ▼                                              │
│  [消费者]                                            │
│   创始人 / 产品经理 / 运营 / 开发                  │
│   定时邮件报表 / 大屏展示 / 手机查看              │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| BI 工具 | Metabase (开源) | 零代码建图表，支持 SQL，部署简单 |
| 备选 | Grafana | 更适合时序/监控数据 |
| 备选 | Apache Superset | 功能最全，学习曲线陡 |
| 数据源 | PostgreSQL 直连 | 初期够用，不需要数据仓库 |
| 数据仓库 | dbt + PostgreSQL | 规模大时做数据转换 |
| 调度 | Cron / Airflow | 定时刷新聚合表 |
| 部署 | Docker Compose | 一键启动 Metabase |

## 实现步骤

### Phase 1：Metabase 部署（1h）
1. Docker Compose 启动 Metabase
2. 连接 PostgreSQL 数据源
3. 自动扫描表结构，生成基础模型
4. 配置访问权限（管理员/查看者）

### Phase 2：指标体系设计（2-3h）
5. 北极星指标定义：周活跃设备数（WAU）
6. 一级指标：日活、新增注册、设备绑定数、OTA 覆盖率
7. 二级指标：功能使用率、BLE 连接成功率、平均使用时长
8. 健康指标：错误率、崩溃率、API 响应时间

### Phase 3：看板搭建（2-3h）
9. 战略看板（CEO 视角）：
   - 用户增长曲线、设备激活趋势、留存率
10. 产品看板（PM 视角）：
    - 功能使用排行、转化漏斗、用户路径
11. 运维看板（开发视角）：
    - 错误率趋势、API 性能、固件版本分布
12. 实时看板（大屏）：
    - 当前在线设备数、今日新增、实时事件流

### Phase 4：自动化报表（1h）
13. 每日晨报：邮件推送关键指标（日活、新增、异常）
14. 周报：自动生成周度趋势对比
15. 异常告警：指标偏离阈值时通知（日活下降 >20%）

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 指标定义不清 | 不同人算出不同数字 | 统一指标口径文档 |
| 查询太慢 | 看板加载 10 秒+ | 预聚合 + 缓存 + 索引优化 |
| 数据不一致 | 看板和实际对不上 | 数据校验 + 口径说明 |
| 看板太多 | 没人看 | 聚焦核心指标，定期清理 |
| 权限混乱 | 敏感数据泄露 | 分组权限 + 数据脱敏 |
| 实时性要求 | 数据延迟用户不满 | 区分实时/准实时/T+1 |
