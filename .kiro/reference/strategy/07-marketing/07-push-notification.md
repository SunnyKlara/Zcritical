# 7. 推送通知系统

## 是什么
向用户手机发送推送消息（通知栏弹出），即使 App 未打开也能触达用户。包括系统通知（OTA 更新、设备异常）和营销通知（活动、新功能），是用户触达的核心通道。

## 为什么需要
- 召回沉默用户：7 天未打开 App → 推送"你的设备有新固件"
- 通知重要事件：OTA 更新可用、设备异常告警、订单状态变更
- 营销触达：新功能上线、限时活动、节日促销
- 提升活跃度：推送是成本最低的用户触达方式（免费）
- 实时性：比短信/邮件更即时，用户看到概率更高

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              推送通知架构                             │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [触发层]                                            │
│   系统事件（OTA/异常）→ 自动触发                   │
│   运营后台 → 手动/定时推送                         │
│   用户行为 → 触发式推送（7天未活跃→召回）         │
│       │                                              │
│       ▼                                              │
│  [推送服务]                                          │
│   目标用户筛选 → 消息模板渲染 → 通道路由          │
│       │                                              │
│       ├─── [iOS] APNs (Apple Push Notification)     │
│       ├─── [Android 海外] FCM (Firebase)            │
│       ├─── [Android 国内] 厂商通道                  │
│       │     ├─ 小米推送 / 华为推送 / OPPO 推送     │
│       │     └─ 极光推送（聚合多厂商）              │
│       │                                              │
│       ▼                                              │
│  [用户设备]                                          │
│   通知栏展示 → 用户点击 → App 打开指定页面        │
│                                                      │
│  [数据层]                                            │
│   送达率 / 点击率 / 转化率 → 优化推送策略         │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 聚合平台 | 极光推送 (JPush) | 国内覆盖好，聚合厂商通道 |
| 备选 | 个推 / 友盟推送 | 类似方案 |
| iOS | APNs | Apple 唯一通道 |
| Android 海外 | FCM | Google 服务 |
| Android 国内 | 厂商通道 (MiPush/HuaweiPush) | 国内无 Google 服务 |
| 消息模板 | 后端模板引擎 | 支持变量替换 |
| 定时推送 | Cron / 消息队列延迟 | 定时发送 |

## 实现步骤

### Phase 1：SDK 集成（2-3h）
1. Flutter 集成极光推送 SDK（`jpush_flutter`）
2. iOS 配置：APNs 证书/Key 上传到极光后台
3. Android 配置：各厂商推送 SDK 集成（极光自动处理）
4. 注册设备 Token → 上报后端关联 user_id

### Phase 2：后端推送服务（2-3h）
5. 推送 API 设计：
   ```
   POST /api/push/send
   {
     "target": {"user_ids": [...]} | {"tags": ["vip"]} | {"all": true},
     "notification": {
       "title": "固件更新",
       "body": "v1.3.0 新增音效包，立即升级",
       "data": {"action": "ota", "version": "1.3.0"}
     },
     "schedule": "2026-06-01T09:00:00+08:00"
   }
   ```
6. 用户分群推送：按标签/属性筛选目标用户
7. 推送记录：每条推送的发送量/送达量/点击量

### Phase 3：App 端处理（1-2h）
8. 前台通知：App 打开时收到推送 → 应用内弹窗/Toast
9. 后台通知：通知栏展示 → 用户点击 → 打开指定页面
10. 深度链接：推送携带 action → App 路由到对应页面
11. 通知权限：首次启动引导用户开启通知权限

### Phase 4：智能推送策略（2h）
12. 用户分群推送：
    - 新用户（注册 3 天内）→ 功能引导
    - 沉默用户（7 天未活跃）→ 召回
    - 活跃用户 → 新功能/活动
13. 推送时机优化：
    - 分析用户活跃时段 → 在最可能打开的时间推送
    - 避免深夜推送（22:00-08:00 静默）
14. A/B 测试标题：同一内容两个标题，看哪个点击率高
15. 频率控制：每用户每天最多 N 条，防止骚扰

### Phase 5：运营后台（1-2h）
16. 推送管理界面：创建/编辑/定时/发送
17. 模板管理：预设常用模板（OTA 通知/活动通知/召回）
18. 数据看板：送达率、点击率、转化率趋势
19. 黑名单：用户关闭通知后不再推送

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 国内 Android 送达率低 | 用户收不到推送 | 必须接厂商通道（极光聚合） |
| 推送太频繁 | 用户关闭通知权限 | 频率控制 + 内容有价值 |
| iOS 权限未授权 | 推送发了但用户看不到 | 引导开启 + 应用内备用通知 |
| 深度链接失败 | 点击推送打开首页而非目标页 | 完善路由 + 冷启动处理 |
| Token 过期 | 推送发送失败 | 定期刷新 Token + 清理无效设备 |
| 推送内容违规 | 被平台封禁 | 遵守各平台内容规范 |

## 与 RideWind 的关系
- 当前状态：无推送能力，用户不打开 App 就失联
- 优先级：P4（用户系统做完后）
- 起步方案：极光推送（国内覆盖好，Flutter SDK 成熟）
- 核心场景：OTA 更新通知、设备异常告警、沉默用户召回
- 成本：极光免费版支持 100 万设备
