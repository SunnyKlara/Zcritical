# 40. 微服务拆分

## 是什么
将单体后端应用拆分为多个独立部署的小服务，每个服务负责一个业务领域。服务间通过 API/消息队列通信，可以独立开发、部署、扩展。

## 为什么需要
- 单体应用膨胀后：改一行代码要全量部署，风险大
- 团队协作：不同人负责不同服务，互不干扰
- 独立扩展：OTA 下载流量大 → 只扩 OTA 服务，不动用户服务
- 技术异构：不同服务可以用不同语言（Go 做高性能，Node 做快速迭代）
- 故障隔离：一个服务挂了不影响其他服务

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              微服务架构                               │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [API Gateway] ← 统一入口                           │
│       │                                              │
│  ┌────┼────┬────────┬────────┬────────┐            │
│  ▼    ▼    ▼        ▼        ▼        ▼            │
│ 用户  设备  OTA     文件     通知     数据          │
│ 服务  服务  服务    服务     服务     服务          │
│  │    │     │       │        │        │            │
│  └────┴─────┴───────┴────────┴────────┘            │
│              │                                       │
│              ▼                                       │
│  [基础设施]                                          │
│   PostgreSQL / Redis / MQ / OSS                     │
│   服务发现 / 配置中心 / 链路追踪                   │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 服务间通信 | HTTP REST / gRPC | REST 简单，gRPC 高性能 |
| 服务发现 | Consul / Nacos / K8s DNS | 动态注册和发现 |
| 配置中心 | Nacos / Apollo / 环境变量 | 集中管理配置 |
| 链路追踪 | Jaeger / Zipkin | 跨服务调用链可视化 |
| 容器编排 | Docker + K8s | 标准化部署 |
| API 契约 | OpenAPI / Protobuf | 服务间接口定义 |

## 实现步骤

### Phase 1：服务边界划分（2h）
1. 按业务领域拆分（DDD 思想）：
   - user-service：注册/登录/资料
   - device-service：设备管理/配置
   - ota-service：固件管理/灰度/下载
   - file-service：文件上传/存储
   - notification-service：推送/短信/邮件
   - analytics-service：数据采集/统计

2. 数据库拆分原则：每个服务独占自己的表/Schema，不跨服务直接查表

### Phase 2：服务通信（2-3h）
3. 同步调用：HTTP REST（简单场景）/ gRPC（高性能场景）
4. 异步通信：消息队列（事件驱动，解耦）
5. 数据一致性：Saga 模式（补偿事务）/ 最终一致性

### Phase 3：基础设施（3-4h）
6. Docker 化：每个服务一个 Dockerfile
7. Docker Compose：本地开发一键启动所有服务
8. 健康检查：`/health` 端点，K8s 探针
9. 配置管理：环境变量 + 配置中心

### Phase 4：可观测性（2h）
10. 统一日志格式：JSON，包含 trace_id
11. 链路追踪：请求经过多个服务时可追踪完整链路
12. 指标采集：Prometheus + Grafana

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 过早拆分 | 复杂度爆炸，开发效率暴跌 | 先单体，痛点明确后再拆 |
| 分布式事务 | 数据不一致 | Saga + 最终一致性 + 补偿 |
| 服务间耦合 | 改一个服务要改三个 | 明确接口契约，版本管理 |
| 网络不可靠 | 调用超时/失败 | 重试 + 熔断 + 降级 |
| 调试困难 | 问题跨多个服务 | 链路追踪 + 统一日志 |
| 运维复杂度 | N 个服务 × 部署/监控/日志 | K8s + CI/CD 自动化 |

## 与 RideWind 的关系
- 当前状态：无后端
- 优先级：P5（用户量到千级再考虑）
- 演进路径：单体 → 模块化单体 → 微服务
- 建议：先用模块化单体（代码分模块但部署为一个进程），等到需要独立扩展时再拆

## 学到什么
- 领域驱动设计（DDD）基础
- 微服务设计原则和反模式
- 分布式系统挑战（CAP、一致性、网络分区）

- 服务间通信模式（同步/异步）
- 可观测性三支柱（日志/指标/追踪）
- 何时该拆、何时不该拆的判断力
