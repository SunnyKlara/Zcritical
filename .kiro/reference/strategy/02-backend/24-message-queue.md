# 24. 消息队列

## 是什么
异步消息中间件。生产者发送消息到队列，消费者异步处理。解耦服务间依赖，削峰填谷，保证最终一致性。

## 为什么需要
- 同步调用链太长 → 用户等待时间长 → 体验差
- 流量尖峰（如 OTA 推送通知）→ 队列缓冲 → 后端不被打垮
- 服务解耦：A 服务不需要知道 B 服务是否在线
- 可靠投递：消息不丢失，失败可重试
- 典型场景：发短信、推送通知、数据同步、日志收集

## 技术架构
```
┌─────────────────────────────────────────────────┐
│              消息队列架构                         │
├─────────────────────────────────────────────────┤
│                                                  │
│  [生产者]                                        │
│   用户注册 → 发送"欢迎短信"消息                │
│   OTA 发布 → 发送"推送通知"消息                │
│   设备上报 → 发送"数据处理"消息                │
│       │                                          │
│       ▼                                          │
│  [消息队列] RabbitMQ / Redis Streams / Kafka    │
│   持久化存储 → 消费确认 → 死信队列             │
│       │                                          │
│       ▼                                          │
│  [消费者]                                        │
│   短信服务 / 推送服务 / 数据处理服务            │
│   失败重试 → 最大重试次数 → 死信队列 → 人工处理│
│                                                  │
└─────────────────────────────────────────────────┘
```

## 技术栈选型
| 场景 | 技术 | 理由 |
|------|------|------|
| 轻量级 | Redis Streams | 已有 Redis，零额外运维 |
| 标准级 | RabbitMQ | 功能完整，社区成熟 |
| 大数据级 | Kafka | 超高吞吐，日志/事件流 |
| 云托管 | 阿里云 RocketMQ | 零运维，事务消息 |
| 延迟队列 | RabbitMQ TTL + DLX | 定时任务（如 30 分钟后检查） |

## 实现步骤

### Phase 1：Redis Streams 入门（2h）
1. 基本操作：XADD 发送、XREADGROUP 消费、XACK 确认
2. 消费者组：多个消费者分摊消息，负载均衡
3. 持久化：消息存储在 Redis，重启不丢失（AOF）

### Phase 2：生产者模式（1-2h）
4. 发送消息：业务代码中 publish 到指定 stream
5. 消息格式：JSON，包含 type/payload/timestamp/retry_count
6. 幂等性：消息带唯一 ID，消费者去重

### Phase 3：消费者模式（2-3h）
7. 消费循环：阻塞读取 → 处理 → ACK
8. 错误处理：处理失败 → 不 ACK → 超时后重新投递
9. 死信队列：重试 N 次仍失败 → 转入死信 → 告警 + 人工处理
10. 优雅关闭：收到 SIGTERM → 处理完当前消息 → 退出

### Phase 4：监控与运维（1h）
11. 积压监控：pending 消息数超阈值 → 告警
12. 消费速率：消费者处理 TPS 统计
13. 死信告警：死信队列有新消息 → 通知开发者

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 消息丢失 | 业务数据不一致 | 持久化 + ACK 机制 |
| 重复消费 | 短信发两次 / 扣款两次 | 消费者幂等设计 |
| 消息积压 | 消费者处理不过来 | 水平扩展消费者 + 告警 |
| 顺序性 | 需要顺序的场景乱序 | 单分区 / 按 key 路由 |
| 大消息 | 队列内存/带宽压力 | 消息只存引用，数据存 OSS |
| 消费者崩溃 | 消息卡在 pending | 超时自动重新投递 |

## 与 RideWind 的关系
- 当前状态：无后端，无消息队列需求
- 优先级：P4（后端服务拆分后才需要）
- 起步场景：OTA 推送通知、短信验证码发送
- 起步方案：Redis Streams（已有 Redis，零额外成本）
