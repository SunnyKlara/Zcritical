# 27. 支付系统

## 是什么
集成在线支付能力，让用户在 App/小程序/网页中完成付款。支持微信支付、支付宝、Apple Pay（iOS 内购）、信用卡等主流支付方式。包括支付、退款、对账的完整闭环。

## 为什么需要
- 电商购买：用户在线购买 RideWind 产品
- App 内购：订阅会员、购买高级音效包/灯效包
- 增值服务：云存储空间、远程诊断服务
- 配件购买：替换零件、周边产品
- 收入基础：没有支付 = 没有商业模式

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              支付系统架构                             │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [客户端]                                            │
│   App → 选择支付方式 → 调起支付 SDK               │
│       │                                              │
│       ▼                                              │
│  [支付网关]（后端）                                  │
│   创建支付单 → 调用支付平台 → 返回支付参数        │
│       │                                              │
│       ├─── [微信支付] App 支付 / JSAPI / Native    │
│       ├─── [支付宝] App 支付 / 手机网站支付        │
│       ├─── [Apple IAP] StoreKit 2                   │
│       └─── [Stripe] 信用卡（海外）                  │
│       │                                              │
│       ▼                                              │
│  [回调处理]                                          │
│   支付平台异步通知 → 验证签名 → 更新订单状态      │
│       │                                              │
│       ▼                                              │
│  [对账系统]                                          │
│   每日下载账单 → 与本地订单比对 → 差异处理        │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 微信支付 | 微信支付 V3 API | 国内主流，App/小程序/H5 |
| 支付宝 | 支付宝开放平台 | 国内第二大 |
| iOS 内购 | StoreKit 2 + 服务端验证 | Apple 强制要求虚拟商品走 IAP |
| 海外 | Stripe | 信用卡/Google Pay/Apple Pay |
| 签名验证 | RSA / HMAC-SHA256 | 防篡改 |
| 对账 | 定时任务 + 差异报表 | 每日自动对账 |
| 支付记录 | PostgreSQL | 完整支付流水 |

## 实现步骤

### Phase 1：微信支付集成（2-3h）
1. 商户号申请 + API 密钥配置
2. 统一下单 API：
   ```
   POST /api/payment/create
   {
     "order_id": "ORD202606010001",
     "amount": 29900,  // 单位：分
     "description": "RideWind Pro 黑色款",
     "method": "wechat_app"  // wechat_app/wechat_jsapi/alipay
   }
   ```
3. App 端调起微信支付 SDK → 用户确认 → 支付完成
4. 异步回调处理：
   ```
   POST /api/payment/notify/wechat
   → 验证签名 → 检查金额 → 更新订单 → 返回 SUCCESS
   ```

### Phase 2：支付宝集成（1-2h）
5. 支付宝开放平台应用创建 + RSA 密钥配置
6. App 支付：生成签名字符串 → App 调起支付宝 SDK
7. 异步通知处理：验证签名 → 更新订单状态

### Phase 3：Apple 内购（2-3h）
8. App Store Connect 配置内购商品：
   - 消耗型：音效包、灯效包（一次性购买）
   - 非消耗型：永久解锁功能
   - 自动续期订阅：会员月/年订阅
9. StoreKit 2 集成（Flutter: `in_app_purchase`）
10. 服务端验证：App Store Server API 验证收据
11. 注意：Apple 抽成 30%（小企业 15%）

### Phase 4：退款系统（1-2h）
12. 退款 API：
    ```
    POST /api/payment/refund
    {
      "order_id": "ORD202606010001",
      "amount": 29900,  // 全额退款
      "reason": "用户申请退款"
    }
    ```
13. 退款到原支付渠道（微信→微信，支付宝→支付宝）
14. 退款状态追踪：申请→处理中→成功/失败
15. Apple 退款：用户通过 Apple 申请，服务端收到通知

### Phase 5：对账系统（1-2h）
16. 每日对账流程：
    - 下载微信/支付宝账单文件
    - 与本地支付记录逐笔比对
    - 差异分类：多付/少付/未到账/重复支付
17. 差异处理：自动补单 / 人工审核
18. 财务报表：日/周/月收入统计、渠道分布、退款率

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 回调丢失 | 用户付了钱但订单未更新 | 主动查询 + 回调重试 + 对账补单 |
| 重复回调 | 订单状态被重复处理 | 幂等设计（已支付的订单忽略重复回调） |
| 签名验证失败 | 伪造支付通知 | 严格验证签名，拒绝无效请求 |
| 金额不一致 | 回调金额与订单金额不符 | 比对金额，不一致则拒绝 |
| Apple 30% 抽成 | 利润大幅减少 | 虚拟商品必须走 IAP，实物可以走自有支付 |
| 掉单 | 用户付款成功但系统无记录 | 定时对账 + 主动查询补单 |

## 与 RideWind 的关系
- 当前状态：无支付能力，通过第三方平台（淘宝）收款
- 优先级：P5（有订阅/电商需求时）
- 起步场景：
  - App 内购买高级音效包/灯效包（Apple IAP + 微信支付）
  - 会员订阅（月/年）
- 注意：iOS 虚拟商品必须走 Apple IAP（30% 抽成）
- 实物商品可以走微信支付/支付宝（无平台抽成）
