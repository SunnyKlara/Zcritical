# 14. 灰度发布/Feature Flag

## 是什么
控制功能的开关和可见范围。新功能先对 1% 用户开放，观察无问题后逐步扩大到 100%。出问题一键关闭，不需要回滚代码。

## 为什么需要
- 降低发布风险：新功能有 bug 只影响少量用户
- A/B 测试：同一功能两种实现，看哪个数据好
- 渐进式发布：按地区/版本/用户群逐步开放
- 紧急开关：线上出问题立即关闭功能，不用发版
- 产品实验：快速验证想法，数据驱动决策

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              Feature Flag 架构                        │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [管理后台]                                          │
│   创建 Flag → 设置规则 → 灰度比例                  │
│       │                                              │
│       ▼                                              │
│  [Flag 服务]                                         │
│   存储规则 → 评估请求 → 返回开关状态              │
│   规则类型：                                        │
│     - 百分比（10% 用户）                            │
│     - 用户白名单（指定 user_id）                   │
│     - 属性匹配（版本>1.2, 地区=华东）             │
│     - 时间窗口（2026-06-01 ~ 2026-06-07）         │
│       │                                              │
│       ▼                                              │
│  [客户端 SDK]                                        │
│   App/后端 查询 Flag 状态 → 条件执行代码           │
│   if (featureFlag.isEnabled("new_ota_ui")) { ... } │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 自建 | Redis + API | 简单场景够用 |
| 开源 | Unleash / Flagsmith | 功能完整，有管理界面 |
| 云服务 | LaunchDarkly / 阿里云 AHAS | 企业级，零运维 |
| 客户端 SDK | HTTP 轮询 / SSE 推送 | 实时获取最新状态 |
| 缓存 | 本地缓存 + 定期刷新 | 减少网络请求 |

## 实现步骤

### Phase 1：简单实现（2h）
1. Redis Hash 存储 Flag：`flag:{name} = {enabled, percentage, whitelist}`
2. API：GET /api/flags/{name}?user_id=xxx → 返回 true/false
3. 评估逻辑：白名单优先 → 百分比哈希（user_id % 100 < percentage）

### Phase 2：管理界面（2h）
4. Flag 列表：名称/描述/状态/规则
5. 创建/编辑 Flag
6. 一键开关（紧急关闭）
7. 操作日志（谁改了什么）

### Phase 3：客户端集成（2h）
8. Flutter SDK：启动时拉取所有 Flag → 本地缓存 → 定期刷新
9. 后端 SDK：中间件注入 Flag 状态
10. 代码中使用：`if (flags.isEnabled('feature_x')) { ... }`

### Phase 4：高级规则（1-2h）
11. 多条件组合：版本 AND 地区 AND 百分比
12. 互斥组：A/B 测试时确保用户只在一个组
13. 生命周期：Flag 过期自动清理（防止代码中残留死 Flag）

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| Flag 残留 | 代码中大量 if/else 死代码 | 定期清理已全量的 Flag |
| 缓存不一致 | 关了 Flag 但客户端还在用 | 短缓存 TTL + 强制刷新接口 |
| 百分比不均匀 | 某些用户总是在实验组 | 用 hash(user_id + flag_name) |
| 测试困难 | 测试环境 Flag 状态不确定 | 测试环境强制覆盖 Flag |
| 性能影响 | 每个请求都查 Flag | 本地缓存 + 批量查询 |

## 与 RideWind 的关系
- 当前状态：无 Feature Flag
- 优先级：P4（有后端 + 一定用户量后）
- 场景：OTA 灰度（先 10% 设备升级）、新 UI 实验、付费功能开关
- 起步：OTA 灰度规则（#1 OTA 中已设计）可以复用 Flag 机制
