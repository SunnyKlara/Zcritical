## 方向一：设备端（01-device）— 产品的物理灵魂

### 本方向本质

设备端是用户**直接触摸、看到、听到**的东西。所有其他方向最终都是为了让这一层的体验更好。对于硬件产品，设备端的质量直接决定退货率和口碑。

### 包含战略项

| # | 名称 | 核心能力 | 行业对标 |
|---|------|----------|----------|
| 1 | OTA | 远程修 bug、推新功能 | DJI/Tesla/小米 IoT 标配 |
| 21 | Secure Boot | 防逆向、防篡改、信任链 | 所有正规 IoT 产品必备 |
| 22 | 低功耗 | 省电、延长寿命 | 电池设备刚需，桌面设备次要 |
| 23 | 传感器融合 | 多传感器数据综合决策 | 无人机/机器人核心 |
| 31 | 音频 DSP | 实时音频处理、声音质感 | 专业音频设备核心 |
| 32 | 显示 UI | LCD 菜单、动画、信息展示 | 消费电子标配 |
| 41 | BLE 深度 | 连接稳定性、吞吐量、多设备 | IoT 设备生命线 |
| 44 | 电机控制 | PWM 精确控制、PID 闭环 | 工业控制/机器人 |
| 45 | 产测 | 出厂自动化测试 | 量产必备 |

     │                         │                │
     │                         └───── 体验 ─────┘

     │                                    ▼
     │                              OTA(#1) 升级

     │                                    ▼

     │                                    │
     │                                    ▼

### 行业视角：设备端开发的三个阶段

**阶段 A：功能实现**（你现在在这里）
- 每个功能能跑就行
- 代码质量次要，先验证效果
- 典型问题：偶发崩溃、内存泄漏、时序竞争

**阶段 B：稳定化**（你即将进入）
- 所有功能在真机上稳定运行 8 小时不崩
- WDT 不触发、内存不泄漏、BLE 不断连
- 典型工作：压力测试、边界条件处理、错误恢复

**阶段 C：产品化**（量产前）
- Secure Boot 锁死、产测流程自动化
- OTA 安全可靠、A/B 分区回滚
- 功耗优化、热管理、EMC 合规

### 关键决策点

1. **OTA 分区策略**：factory 模式 vs A/B OTA 模式（你已决策用 A/B，正确）
2. **音频架构**：MP3 播放 vs 实时合成（你已决策用 4 层合成，正确）
3. **BLE vs WiFi 主通道**：BLE 控制 + WiFi 音频（你已决策，正确）
4. **Secure Boot 时机**：产品稳定后再锁（正确，过早锁会拖慢开发）
5. **产测方案**：串口自动化 vs 专用治具（取决于量产规模）

### 你的现状 vs 行业标准

| 项 | 你的现状 | 行业标准 | 差距 |
|----|----------|----------|------|
| OTA | ✅ WiFi OTA 已实现 | A/B + 签名 + 回滚 | 缺签名和强制回滚 |
| Secure Boot | ❌ 未启用 | 量产前必须 | 产品稳定后做 |
| 音频 DSP | ✅ 变速率+混音 | +EQ+压缩+限幅 | 缺效果链 |
| BLE | ✅ 能连能通信 | 稳定8h不断 | 偶发断连/重启 |
| 显示 UI | ✅ 菜单系统完整 | 流畅动画+多语言 | 够用 |
| 产测 | ❌ 无 | 自动化测试脚本 | 量产前做 |

## 方向二：后端服务（02-backend）— 在线能力的地基

### 本方向本质

后端是让产品从"单机设备"变成"联网服务"的基础设施。没有后端，设备就是一个孤岛——数据不能云同步、用户不能跨设备、你不能远程触达用户。

### 包含战略项

| # | 名称 | 核心能力 | 何时需要 |
|---|------|----------|----------|
| 2 | 用户系统 | 注册登录、身份认证 | 任何在线功能的前提 |
| 3 | 设备管理 | 设备注册、绑定、状态 | 知道"谁的设备" |
| 4 | 数据存储 | PostgreSQL/Redis | 所有数据的家 |
| 9 | API 网关 | 统一入口、认证、限流 | 多服务时的门卫 |
| 24 | 消息队列 | 异步解耦、削峰填谷 | 高并发时的缓冲 |
| 33 | 文件存储 | 图片/固件/音频文件 | 任何文件上传下载 |
| 40 | 微服务 | 服务拆分、独立部署 | 团队>5人时 |

### 运转逻辑

```
用户打开App
    │
    ▼
API网关(#9) ─── 认证 ─── 用户系统(#2)
    │                         │
    ├── 查设备 ──→ 设备管理(#3) ──→ 数据存储(#4)
    │                                      ▲
    ├── 下载固件 ──→ 文件存储(#33) ────────┘
    │
    ├── OTA推送 ──→ 消息队列(#24) ──→ 设备
    │
    └── 规模化后 ──→ 微服务(#40) 拆分
```

### 行业视角：后端架构的演进路径

**Level 0：无后端**（你现在）
- App 数据纯本地
- 固件文件放 CDN 静态托管
- 适合：<500 用户，单设备绑定

**Level 1：轻量后端**（你的下一步）
- 一台服务器跑所有服务（monolith）
- 用户系统 + 设备绑定 + OTA 版本管理
- 技术：Node.js/Go + PostgreSQL + Redis
- 成本：阿里云 ECS 99 元/月
- 适合：500-5000 用户

**Level 2：标准后端**
- 服务开始拆分（用户服务、设备服务、OTA 服务）
- 加入消息队列处理异步任务
- API 网关统一管理
- 适合：5000-50000 用户

**Level 3：规模化后端**
- 微服务架构 + 容器编排
- 数据库读写分离 + 分库分表
- 全球多区域部署
- 适合：>50000 用户

### 关键决策点

1. **语言选择**：Node.js（生态丰富、你熟悉 JS）vs Go（性能好、部署简单）vs Python（AI 集成方便）
2. **数据库**：PostgreSQL（关系型、功能全）vs MongoDB（文档型、灵活）— 推荐 PG
3. **部署方式**：ECS 自管 vs Serverless（函数计算）— 初期 Serverless 省心
4. **何时上后端**：当你需要"用户换手机不丢数据"或"远程推送通知"时
5. **自建 vs 第三方**：Firebase/Supabase 可以零代码搞定 Level 1

### 你的现状 vs 行业标准

| 项 | 你的现状 | 最小可行 | 完整形态 |
|----|----------|----------|----------|
| 用户系统 | ❌ 无 | 手机号+验证码 | OAuth+多端同步 |
| 设备管理 | ❌ 无 | SN绑定+状态 | 多设备+转让+共享 |
| 数据存储 | 本地 only | PostgreSQL | PG+Redis+OSS |
| API 网关 | ❌ 无 | Nginx 反代 | Kong/APISIX |
| 文件存储 | CDN 静态 | 阿里云 OSS | OSS+CDN+权限 |

## 方向三：IoT 云（03-iot-cloud）— 设备与云端的神经通道

### 本方向本质

IoT 云层解决的核心问题是：**设备不在用户手边时，你怎么知道它的状态、怎么控制它？** 这是从"本地控制"到"远程智能"的跨越。

### 包含战略项

| # | 名称 | 核心能力 | 行业对标 |
|---|------|----------|----------|
| 5 | MQTT | 设备-云端双向实时通信 | AWS IoT Core / 阿里云 IoT |
| 6 | 设备影子 | 设备离线时的状态缓存 | AWS Device Shadow / Azure Twin |

### 运转逻辑

```
设备在线时：
  设备 ──MQTT──→ Broker ──→ 后端（实时处理）
  后端 ──MQTT──→ Broker ──→ 设备（实时控制）

设备离线时：
  后端想改设备状态 ──→ 写入"设备影子"(#6)
  设备重新上线 ──→ 从影子同步最新期望状态 ──→ 执行

你的特殊情况（设备无 WiFi）：
  设备 ──BLE──→ App ──MQTT──→ Cloud（App 做网关代理）
```

### 行业视角：IoT 通信的三种模式

**模式 A：设备直连云端**（WiFi/4G 设备）
- 设备自己连 MQTT Broker
- 优点：实时性好、不依赖手机
- 缺点：需要 WiFi 配网、功耗高
- 代表：智能家居（小米/涂鸦）

**模式 B：App 网关代理**（BLE 设备，你的情况）
- 设备通过 BLE 连 App，App 转发到云端
- 优点：设备不需要 WiFi 模块、省电
- 缺点：App 不开就断联、实时性差
- 代表：手环/手表（小米手环）、蓝牙锁

**模式 C：混合模式**（你的未来）
- 日常通过 BLE+App 控制
- WiFi 可用时设备直连云端（OTA、诊断）
- 优点：兼顾省电和远程能力
- 代表：高端 IoT（Sonos、Philips Hue）

### 关键决策点

1. **Broker 选型**：自建 EMQX vs 云托管（阿里云 IoT Core）— 初期用云托管省心
2. **设备认证**：一机一密 vs 证书认证 — 一机一密简单，证书更安全
3. **数据格式**：JSON（可读）vs Protobuf（省带宽）vs CBOR（二进制 JSON）
4. **QoS 级别**：QoS 0（最多一次）vs QoS 1（至少一次）— 控制命令用 QoS 1
5. **何时需要**：当你想"不通过 App 就能知道设备状态"或"远程推送 OTA"时

### 你的现状 vs 行业标准

| 项 | 你的现状 | 最小可行 | 完整形态 |
|----|------|----------|----------|
| 设备通信 | 纯 BLE 本地 | App 代理上报 | MQTT 直连+影子 |
| 远程控制 | ❌ 不可能 | 通过 App 转发 | 云端直接下发 |
| 离线状态 | 不知道 | App 上报最后状态 | 设备影子自动同步 |
| 设备在线检测 | ❌ 无 | App 心跳上报 | MQTT Will Message |

### 对你的实际意义

你的设备是**桌面摆件**，不是随身设备。用户大部分时间在设备旁边，BLE 直连就够了。IoT 云层对你的价值主要是：
- **远程 OTA 触发**：不需要用户打开 App 就能推送升级通知
- **设备健康监控**：知道哪些设备频繁重启/报错
- **售后诊断**：用户报问题时远程拉日志

这不是 P0，但在用户量过千后会变得重要。

## 方向四：前端/多端（04-frontend）— 用户看到的一切

```

**Sonos**：

5. **i18n 时机**：准备出海时做，不是现在

## 方向五：DevOps（05-devops）— 开发效率的倍增器

### 本方向本质

DevOps 不是一个功能，是一种**让你更快、更安全地交付代码**的能力。它的价值不是用户直接感知的，而是让你从"每次发版心惊胆战"变成"随时可以自信发版"。

### 包含战略项

| # | 名称 | 核心能力 | 何时需要 |
|---|------|----------|----------|
| 10 | CI/CD | 自动构建、测试、部署 | 第一天就该有 |
| 11 | 容器化 | Docker 打包、环境一致 | 有后端服务时 |
| 12 | 日志监控 | 知道线上发生了什么 | 有用户在用时 |
| 13 | 测试 | 自动化验证不出错 | 代码超过 1000 行时 |
| 14 | Feature Flag | 功能开关、灰度发布 | 用户>1000 时 |
| 29 | IaC | 基础设施即代码 | 多环境/多服务器时 |
| 30 | Service Mesh | 服务间通信治理 | 微服务>5个时 |

### 运转逻辑

```
开发者写代码
    │
    ▼
git push ──→ CI/CD(#10) 自动触发
                │
                ├── 编译检查（能不能编译通过）
                ├── 测试(#13)（功能有没有坏）
                ├── 代码质量（lint、覆盖率）
                │
                ▼
            构建产物（APK / 固件 bin / Docker 镜像）
                │
                ├── Feature Flag(#14) 控制谁能看到新功能
                ├── 容器化(#11) 打包后端服务
                │
                ▼
            部署到生产环境
                │
                ├── IaC(#29) 管理服务器配置
                ├── Service Mesh(#30) 管理服务间通信
                │
                ▼
            日志监控(#12) 观察线上状态
                │
                └── 出问题 → 回滚 → 修复 → 重新走流程
```

### 行业视角：DevOps 成熟度模型

**Level 0：手动一切**
- 本地编译 → U盘拷贝 → 手动安装
- 出了问题不知道哪个版本引入的
- 你 3 个月前在这里

**Level 1：基础自动化**（你现在在这里）
- CI 自动编译+检查
- CD 自动构建发布包
- 手动触发部署
- 有基本的版本管理

**Level 2：持续交付**
- 每次合并自动部署到测试环境
- 一键部署到生产

## 方向六：数据（06-data）— 用数据代替直觉做决策

### 本方向本质

数据层的核心问题是：**你怎么知道产品做得好不好？** 没有数据，所有决策都是猜测。有了数据，你能回答"哪个功能最受欢迎"、"用户在哪一步流失"、"这次改动有没有让体验变好"。

| 34 | 用户行为分析 | 路径分析、漏斗、留存 | 用户为什么走了？ |
| 46 | A/B 测试 | 对照实验、科学验证 | 哪个方案更好？ |

### 运转逻辑

```

    │
    ▼
埋点(#15) 采集事件 ──→ 数据管线 ──→ 数据仓库
                                        │
                    ┌───────────────────┼───────────────────┐

            BI看板(#16)        用户行为(#34)         A/B测试(#46)

                                        │
                                        ▼
                                    产品决策

### 行业视角：数据驱动的四个层次

**Level 0：盲飞**（你现在）
- 不知道有多少活跃用户
- 不知道哪个功能被用得最多
- 决策靠直觉和用户微信反馈

**Level 1：基础度量**（你的下一步）
- 知道 DAU/MAU、功能使用率、崩溃率
- Firebase Analytics 免费搞定
- 工作量：半天集成 + 1 天埋点

**Level 2：行为分析**
- 知道用户路径（打开App→连接→调灯→退出）
- 知道漏斗转化（100人下载→60人注册→30人绑定→20人活跃）
- 知道留存曲线（D1=60%, D7=30%, D30=15%）

**Level 3：实验驱动**
- 每个改动都跑 A/B 测试
- 数据证明"新方案比旧方案好 15%"才上线
- 需要足够的用户量（每组>1000 才有统计意义）

### 关键决策点

1. **埋点方案**：Firebase Analytics（免费、Flutter 原生）vs 友盟（国内合规）
2. **自建 vs 第三方**：初期用 Firebase，用户过万后考虑自建（ClickHouse）
3. **埋什么**：核心漏斗（下载→注册→绑定→活跃→付费）+ 功能使用率
4. **隐私合规**：GDPR/个保法要求用户同意后才采集，App 需要隐私弹窗
5. **A/B 测试时机**：用户<1000 时样本不够，做了也没统计意义

| A/B 测试 | ❌ 无 | 不需要 | Firebase Remote Config |

### 对你的实际意义

数据是**阶段 2 的基础设施**。现在你用户少，微信群就能感知用户需求。但当用户过百后，你需要数据来回答：

- 优点：简单直接
- 缺点：增长靠卖更多设备，天花板明显
- 代表：传统消费电子

- 设备是平台，内容是持续收入
- 代表：Peloton（健身车+课程订阅）、Sonos（音箱+流媒体）
- 你的对应：付费音效包、灯效方案、车型数据包

**模式 D：硬件 + SaaS 订阅**
- 设备功能按月付费解锁
- 代表：Ring（门铃+云存储月费）、Tesla（FSD 订阅）
- 你的对应：Pro 会员（高级灯效、音频投射、多设备同步）
- ⚠️ 风险：硬件用户对订阅抵触大，需要提供明确额外价值

**推荐路径**：A → B+C → 观察数据再决定是否 D

### 关键决策点

1. **销售渠道**：淘宝（现在）→ 自建商城（月销>500台时）→ 海外 Amazon
2. **定价策略**：成本×2.5~3 = 零售价（留出平台佣金和营销费用）
3. **付费内容**：音效包（¥9.9）、灯效方案（¥4.9）、车型数据包（¥19.9）
4. **订阅模式**：谨慎，硬件用户对"买了还要付费"很敏感
5. **支付集成**：微信支付+支付宝（国内）、Stripe（海外）

| 项 | 你的现状 | 最小可行 | 完整形态 |

|---|------|----------|-----------|
| 28 | 合规 | 法律法规遵从 | 不被下架/罚款 |

### 本方向本质

安全合规是**保护性投资**——做了不会让你多赚钱，但不做可能让你赔很多钱或者产品被下架。

### 行业视角：IoT 产品的合规清单

```
硬件合规（出厂前）：
├── CE 认证（欧洲市场）
├── FCC 认证（美国市场）
├── CCC 认证（中国市场，部分品类）
├── RoHS（有害物质限制）
└── WEEE（电子废弃物回收标识）

软件合规（上架前）：
├── 隐私政策（App Store/Google Play 强制要求）
├── 用户协议
├── GDPR（欧洲用户数据保护）
├── 个人信息保护法（中国）
├── COPPA（如果有未成年用户）
└── App Store 审核指南遵从

通信合规：
├── 蓝牙 SIG 认证（使用蓝牙 Logo 需要）
├── WiFi 联盟认证（可选）
└── 无线电型号核准（中国 SRRC）
```

### 关键决策点

1. **隐私政策**：App 上架前必须有，可以用模板生成
2. **数据存储**：国内用户数据存国内服务器（个保法要求）
3. **蓝牙认证**：如果要在包装上印蓝牙 Logo，需要 BT SIG 会员（$2500/年）
4. **CE/FCC**：出口必须，国内销售看品类是否强制
5. **时机**：隐私政策现在就要，硬件认证量产前做

### 你的现状

- 隐私政策：App 上架前需要准备
- 硬件认证：取决于销售市场和渠道要求
- 数据合规：目前无后端，暂不涉及
- 优先级：隐私政策 P2（上架前），硬件认证 P3（出口时）

## 总结：CTO 视角的三个核心认知

### 1. 这 50 项不是 TODO List，是一张地图

你不需要做完 50 项。你需要的是：在做任何一个决策时，知道它在地图上的位置，知道它会影响什么、被什么影响。

### 2. 每个阶段只做那个阶段的事

阶段 1 做阶段 3 的事 = 过度工程 = 浪费时间
阶段 3 才做阶段 1 的事 = 技术债爆炸 = 重写

你现在在阶段 1→1.5 的过渡期。该做的事：
- 设备稳定（BLE 不断、不重启）
- App 体验打磨（iOS 上架）
- 基础数据（Firebase 埋点）

### 3. 单人开发的核心约束是时间，不是技术

50 项你都知道怎么做（文档已经写得很清楚了）。瓶颈是：一个人一天只有 8 小时，一周只有 5 个工作日。所以每一项的 ROI（投入产出比）才是真正的决策依据：

- 高 ROI：OTA（1天投入，永久受益）✅ 已做
- 高 ROI：CI/CD（1天投入，每次发版省 2 小时）✅ 已做
- 高 ROI：Firebase 埋点（半天投入，永久有数据）← 下一个
- 低 ROI：微服务架构（5天投入，现在用不上）← 远期
- 低 ROI：Service Mesh（3天投入，你只有 1 个服务）← 可能永远不需要
