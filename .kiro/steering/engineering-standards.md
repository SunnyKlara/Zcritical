# 工程标准 — 500+ 用户级产品的底线规则

> 本文件定义不可违反的工程规则。不是建议，是强制执行。
> 违反任何一条都可能导致用户受损或项目不可维护。

---

## 第一原则：已发布的契约不可破坏

**定义**：任何写死在已发布 APK/固件中的东西，都是对现有用户的承诺。

包括但不限于：
- 版本检测 URL（APP 启动时请求的地址）
- JSON 字段名（APP 解析远程配置的 key）
- BLE 协议命令格式（固件解析 APP 发来的字符串）
- BLE Service/Characteristic UUID
- 服务器 API 路径

**规则**：
1. 重构时，旧接口必须保留或做兼容层，直到确认所有旧版本用户已升级
2. 新增字段/命令可以，删除/改名不行
3. 如果必须破坏兼容性，必须：
   - 在 CHANGELOG 中标注 BREAKING CHANGE
   - `forceUpdate: true` 强制旧版本升级
   - 确认旧版本能收到升级提示（否则就是把用户锁死在旧版本）

**v1.2.1 教训**：升级系统重构时改了版本检测 URL 路径，旧 APP 永远 404，500 用户无法自动升级。

## 第二原则：一个版本只做一件事

**规则**：
- 每个版本号（tag）对应一个主要改动
- 不允许在一个版本里同时做：架构重构 + 新功能 + Bug 修复 + 品牌改名
- 出了问题能精确定位是哪个改动导致的

**版本节奏**：
```
v1.2.2 — 升级系统修复（只改升级相关代码）
v1.2.3 — 首页改版（只改导航和首页 UI）
v1.2.4 — BLE 稳定性（只改连接相关逻辑）
v1.3.0 — 新功能（能力协商系统）
```

**例外**：紧急 hotfix 可以在任何版本上叠加，但必须是最小改动。

---

## 第三原则：发版前必须真机验证

**强制验证清单**（每次发版前，无例外）：

```
□ 1. 全新安装（卸载旧版 → 安装新 APK）
□ 2. 打开 APP → 首页正常显示
□ 3. 扫描 → 连接设备 → 进入控制页面
□ 4. 核心操作：调速/调灯/调风扇 各一次
□ 5. 断开蓝牙 → 重连成功
□ 6. 杀掉 APP → 重新打开 → 自动重连或手动连接
□ 7. 升级提示：修改 app_version.json 指向更高版本 → 重启 APP → 弹窗出现
□ 8. 返回/退出 APP 不崩溃
```

**时间成本**：10 分钟。省掉这 10 分钟 = 500 用户遇到问题。

**如果没有硬件在手**：至少完成 1-2-6-8（不需要硬件的部分），并在 CHANGELOG 标注"未经实机验证"。

## 第四原则：双平台不能只跑一个

**当前状态**：iOS 从未在真机上运行过。这是定时炸弹。

**最低要求**：
1. 每个版本发布前，`flutter build ios --release` 必须能编译通过
2. CI 中 iOS 构建必须是 required check（失败则不能发版）
3. 每月至少一次 iOS 真机验证（TestFlight 内测）

**iOS 启动路径**（一次性，按顺序执行）：
1. Apple Developer 账号注册（688 元/年）
2. Mac 上 `flutter build ios` 确认编译通过
3. Xcode 配置签名 → Archive → 上传 TestFlight
4. CI 加入 iOS 构建 job（已有框架，需要补 secrets）
5. 每次 tag 自动构建 iOS + Android

**不做 iOS 的后果**：代码里积累 Android-only 假设（文件路径、权限模型、后台行为），等到要做时发现要改 50 个地方。

## 第五原则：测试是安全网，不是负担

**当前测试覆盖**：
- 协议解析：51 个测试 ✅
- UI/业务逻辑：0 个测试 ❌
- 固件：0 个测试 ❌

**最低要求（现阶段）**：
1. 协议层 100% 覆盖（新命令必须有测试，已做到）
2. 升级系统有测试（版本比较逻辑、JSON 解析、URL fallback）
3. BLE 状态机有测试（状态转换、重连逻辑）

**不需要做的**：
- 不需要 UI 测试（投入产出比太低）
- 不需要 100% 覆盖率（没意义）
- 不需要 mock 硬件（太复杂）

**测试策略**：只测"出错代价高"的逻辑：
- 协议解析（错了 = 设备失控）
- 版本比较（错了 = 升级断裂）
- 状态机转换（错了 = 连接异常）

## 第六原则：监控和反馈闭环

**500 用户时必须有的**：
1. 崩溃上报（Sentry）— 知道 APP 在用户手机上崩了
2. 用户反馈渠道 — 知道用户遇到了什么问题
3. 版本分布统计 — 知道多少人还在旧版本

**当前状态**：
- Sentry：框架已接入，DSN 已配置 ✅
- 反馈渠道：设置页邮箱 ✅
- 版本分布：❌ 无（后续可通过 Sentry 的 release 功能获取）

**万级用户时需要加的**：
- APM（性能监控）— 知道 APP 卡不卡
- 灰度发布数据 — 知道新版本有没有问题
- 服务器监控 — 知道下载服务是否正常

## 第七原则：服务器是生命线

**当前架构**：单台阿里云轻量服务器（47.107.143.4）

**必须保证的**：
1. APK 下载地址永远可用（挂了 = 用户无法升级）
2. 版本检测 JSON 永远可用（挂了 = 升级提示消失）
3. HTTPS 证书不过期（过期 = Android 拒绝连接）

**运维最低要求**：
- Let's Encrypt 证书自动续期（certbot renew cron job）
- nginx 配置备份（本地保留一份）
- 服务器 SSH 密钥安全存储（不放在公开仓库）
- 每次发版后验证下载链接可用（CI 已做 ✅）

**扩展路径**（用户量增长时）：
- 1000+ 用户：加 CDN（阿里云 OSS + CDN，月费 <50 元）
- 5000+ 用户：多地域部署或对象存储
- 10000+ 用户：考虑应用商店分发（减轻自建服务器压力）

## 第八原则：代码质量底线

**不可妥协的**：
1. `flutter analyze` 零 error（warning 可以有，error 不行）
2. `idf.py build` 零 error 零 warning
3. 协议测试全部通过
4. 单文件不超过 1000 行（ESP32 的 main.c 例外）
5. 新增公开 API 必须有注释说明用途和参数

**可以妥协的**：
- 代码风格不完美（有时间再整理）
- 部分 warning（info 级别的 lint）
- 测试覆盖率不高（重点覆盖关键路径）

---

## 第九原则：变更管理

**每次代码变更必须回答三个问题**：
1. 这个改动会影响已发布版本的用户吗？
2. 这个改动需要固件和 APP 同时升级吗？
3. 如果这个改动有 bug，用户会遇到什么问题？能回退吗？

**高风险改动清单**（必须额外谨慎）：
- 升级系统相关代码
- BLE 协议格式变更
- 导航流程变更（用户打开 APP 看到的第一个页面）
- 服务器地址/路径变更
- 签名/包名变更

---

## 第十原则：技术债务管理

**不是说不能欠债，而是必须记录并有计划偿还。**

技术债务记录格式：
```
| 债务 | 风险等级 | 触发条件 | 偿还计划 |
|------|----------|----------|----------|
| iOS 从未真机验证 | 高 | iOS 用户安装后崩溃 | v1.3.0 前完成 |
| R8 minification 关闭 | 低 | APK 体积偏大 | 用户量破千后处理 |
| 旧测试 7 个失败 | 中 | 重构时不知道是否引入回归 | 下次碰到相关代码时修 |
```

**规则**：
- 新增技术债务时必须记录到 CONTINUATION_GUIDE.md
- 高风险债务必须在 2 个版本内偿还
- 中风险债务必须有明确的触发条件（什么时候必须处理）

## 执行方式

这些规则通过以下机制强制执行：

1. **AI 自检**：每次写代码前检查是否违反上述原则
2. **发版 hook**：tag push 前自动运行验证清单
3. **CI 门禁**：analyze + test 不通过不能发版
4. **对话结束自检**：是否引入了新的技术债务

**如果 AI 发现用户的需求会违反上述原则，必须明确指出风险，不能默默执行。**
