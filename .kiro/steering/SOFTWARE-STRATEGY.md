# 软件战略与学习路径

> 本文档是 RideWind 产品软件层面的长期战略参考。
> 它不是 to-do list，而是一份"思考框架 + 学习地图"。
> 当焦虑"是不是漏了什么"的时候，回来对一遍这些维度。
> 每个月回看一次，更新"我现在在哪个阶段"。

---

## 文档结构

- **Part A**：产品软件演进的 12 个维度（产品本身从 1 台到 10 万台用户）
- **Part B**：公司软件版图的 16 个维度（作为软件总负责人需要思考的全部）
- **Part C**：学习路径（每个维度的技术栈、推荐学习顺序、AI 协作策略）
- **Part D**：当前阶段自检表（每月更新一次）
- **Part E**：30 项技术系统清单（除 App 外随产品规模需要跟上设计的所有技术）

---

# Part A：产品软件演进的 12 个维度

随着用户量增长（1 → 100 → 1000 → 1 万 → 10 万），软件层面会暴露的问题。

## A1. 代码本身的可持续性
**问题本质**：今天 AI 写的代码，明天能不能被人接手、被修改、被扩展？

**会暴露的具体问题**：
- AI 写的代码风格不一致（每次 prompt 上下文不同）
- 没有统一的架构思想，整体像拼接的
- 注释和文档跟不上代码迭代
- 边界条件处理不全，bug 延迟爆发
- 招人时新人无法理解项目设计原则

**什么时候爆**：需要找人接手或大改某个模块时。
**今天能做的**：建立"架构宪法"——分层原则、命名规范、错误处理方式。让 AI 每次都遵守。

---

## A2. 协议与数据格式的"长期债务"
**问题本质**：BLE 协议、固件 / App 通信格式、存储格式——一旦发出去就改不回来。

**会暴露的具体问题**：
- 老固件设备升级新协议成砖
- 数据格式改了老数据迁移要写一堆兼容代码
- 命令字段没加版本号，扩展只能开新通道
- 协议没加密 / 签名，被逆向后山寨满天飞

**什么时候爆**：第一个公开版本之后的每次大版本迭代。
**今天能做的**：协议第一字段就是版本号；保留扩展位；想清楚加密策略。

---

## A3. 可观测性
**问题本质**：用户说"设备坏了"，你怎么知道是怎么坏的？

**会暴露的具体问题**：
- 用户反馈"连不上"，不知道是 BLE 没扫到、配对失败还是协议握手错误
- 设备死机重启，不知道死之前在做什么
- App 崩溃没有崩溃报告，只能靠用户截图
- 100 个用户出问题不知道共同特征
- 售后变成黑盒

**什么时候爆**：从有真实用户的第一天开始。
**今天能做的**：固件埋日志（先存本地）；App 接 Crashlytics；每台设备唯一 ID；关键事件留痕。

---

## A4. 发布与回滚机制
**问题本质**：按下"发布"按钮，如果出错能不能撤回？

**会暴露的具体问题**：
- 新固件推送给所有用户，bug 让 1000 台设备同时变砖
- App 强制更新发现重大问题，应用商店审核要 3 天
- 没有灰度发布，每次更新都是全员豪赌
- 没有回滚机制，回滚要等下个版本审核

**什么时候爆**：第一次大版本更新出 bug 时。
**今天能做的**：固件 OTA 双分区设计；App 渐进式发布；服务端配置开关化。

---

## A5. 依赖与供应链
**问题本质**：你的产品依赖一堆别人的东西，他们出问题你跟着出问题。

**会暴露的具体问题**：
- 第三方 SDK 停服或涨价被绑架
- 苹果改隐私政策导致 App 下架
- Google 改 BLE 权限规则导致 App 不能扫描
- 开源库爆高危漏洞你不知道
- 芯片停产、固件强依赖某芯片特性，移植成本巨大
- AI 服务变 API 或停服，依赖功能全废

**什么时候爆**：随时，且无法预测。
**今天能做的**：核心功能不依赖单一外部服务；关键 SDK 加抽象层；定期检查依赖安全公告。

---

## A6. 安全
**问题本质**：用户和数据多到一定规模，你就是攻击目标。

**会暴露的具体问题**：
- BLE 协议没加密，能控制路人设备
- 固件没签名验证，被刷自制固件破解
- App 硬编码 API key，反编译后泄露
- 后端没权限校验，A 用户能看到 B 用户数据
- 用户密码明文存储或弱哈希
- 没防爬虫 / 防刷接口，被打爆服务器
- 一次数据泄露事件，企业基本结束

**什么时候爆**：用户量到一定规模时成为有目标的攻击对象。
**今天能做的**：所有传输 HTTPS / 加密 BLE；密钥不进代码；API 全部鉴权；考虑固件签名。

---

## A7. 合规
**问题本质**：用户量大了法律就盯上你。

**会暴露的具体问题**：
- 国内：等保、个人信息保护法、数据出境
- 海外：GDPR、CCPA、苹果 ATT
- 蓝牙产品要 BQB / CE / FCC / SRRC 认证
- 涉及音频要考虑版权
- App 必须有隐私政策、用户协议、未成年人保护
- 数据留存：哪些能存、存多久、怎么删

**什么时候爆**：出海时、被监管检查时、被用户投诉时。
**今天能做的**：架构设计上"用户数据可删除、可导出"做成基础能力。

---

## A8. 可测试性
**问题本质**：怎么知道改一行代码不会把别的地方搞坏？

**会暴露的具体问题**：
- 没自动化测试，每次发布手动测，越改越不敢改
- 改 A 搞坏 B，测试漏掉
- 嵌入式难测，因为依赖硬件
- AI 写代码容易"修一个 bug 引入两个 bug"，没测试发现不了

**什么时候爆**：代码量到一定规模、改动频率变高时。
**今天能做的**：核心协议层、状态机、计算逻辑写单元测试；硬件部分用模拟器或抽象层测。

---

## A9. 组织与流程
**问题本质**：从 1 个人到 5 个人到 20 个人，软件组织方式会三次推倒重来。

**会暴露的具体问题**：
- 多人协作代码冲突频繁，没有 code review
- 没有 CI/CD，发布手动
- 文档断层，新人入职像考古
- 决策权不清晰
- 你从写代码变管理者，技能完全两套

**什么时候爆**：第二个全职软件加入团队那天。
**今天能做的**：现在就规范 Git 分支模型；写技术决策文档；提前学一点工程管理。

---

## A10. 成本
**问题本质**：用户量上去后，每多一个用户都在花你钱。

**会暴露的具体问题**：
- 云服务按量收费，10 万用户成本是 100 用户的 1000 倍
- 推送、短信、AI 调用都按次数花钱
- 第三方 SDK 免费额度撑到一定用户数就要付费
- 流量、存储、带宽指数级增长

**什么时候爆**：用户增长但收入跟不上时。
**今天能做的**：每个外部服务算清"边际成本"——多一个用户多花多少钱。

---

## A11. 用户体验的一致性
**问题本质**：硬件、固件、App、官网、客服——任何一环断了都是事故。

**会暴露的具体问题**：
- App 显示状态和设备实际状态不一致
- 不同设备 / App 版本组合行为不一致
- 故障时只看到"未知错误"
- 没有内置诊断工具
- 客服没有工具看用户设备状态

**什么时候爆**：用户基数足够大时长尾问题处理不过来。
**今天能做的**：错误码体系化；App 内置"诊断"功能；客服工具同步规划。

---

## A12. 知识沉淀与团队记忆
**问题本质**：所有"为什么这么做"的决定，是不是只活在你脑子里？

**会暴露的具体问题**：
- 半年前为什么选 BLE 而不是 WiFi 记不清
- 某个 workaround 是为了绕过什么 bug 没人知道
- 你休假/生病没人能动这个项目
- 团队扩张时新人重复踩你踩过的坑
- AI context 一断它也不知道项目历史

**什么时候爆**：想休息、招人、外包某块时。
**今天能做的**：决策记录（ADR）；CONTINUATION_GUIDE.md 已经在做。

---

# Part B：公司软件版图的 16 个维度

作为软件总负责人，需要思考的全景。

## B1. 平台形态的取舍
**核心问题**：App / 小程序 / 桌面端 / 网站 / 独立站，每个的存在理由是什么？

每个平台的真实定位：
- **App**：高频、后台运行、需要硬件权限（BLE/相机）
- **微信小程序**：低频但需要轻触达，扫码即用，售前 / 售后短交互
- **桌面端**：批量管理、专业用户、复杂界面
- **官网**：品牌门户，SEO 主战场
- **独立站**：电商功能，卖货收款发货客服
- **公众号 / 视频号**：内容触达，不是产品

判断原则：每个平台问"如果不做，损失什么？" 答不出就不做。

---

## B2. 技术栈的统一性策略
**核心问题**：一个人维护多端，绝不能每个端用一套技术栈。

可行思路：
- 核心层抽出来：用户认证、API、业务规则做成独立包
- 跨端框架优先：App 用 Flutter、桌面端用 Flutter Desktop / Tauri、网站用 Next.js
- 小程序是例外：微信原生或 Taro / uni-app
- 后端只有一套，所有端调同一组 API

---

## B3. 账号与身份系统
**核心问题**：用户在 App / 小程序 / 独立站登录的是同一个账号吗？

核心抽象：
- 统一用户中心，一个用户一个 ID
- 多种登录方式映射到同一 ID（手机号、微信、Apple ID、邮箱）
- 设备绑定关系：一个用户 N 个设备，多个用户可共享一个设备

---

## B4. 数据所在地——单一事实来源
**核心问题**：用户的设置 / 订单 / 设备状态——谁是权威？

权威归属：
- 设备状态：以设备为权威（本地是真相，云端备份）
- 用户数据：以云端为权威
- 订单 / 交易：以独立站后端为权威
- 永远不要让两个端各自独立存储同一类数据

---

## B5. 渠道策略——独立站 vs 第三方平台
**核心问题**：卖货走自己独立站还是淘宝 / 京东 / 抖音？

- **独立站**：所有数据是你的，但要自建电商系统、自己引流、自己风控
- **第三方平台**：不需要电商技术栈，但用户数据不归你，平台规则变了你跟着变

早期建议：第三方平台为主 + 简单官网（不带电商）。品牌起来再做独立站。

---

## B6. 内容系统（CMS）
**核心问题**：官网产品介绍、教程、FAQ、新闻，改内容不应该改代码。

选项：Strapi（开源自部署）、Sanity / Contentful、Notion（也能当 CMS）、WordPress。
原则：哪怕只有 5 个页面也用 CMS。

---

## B7. 流量与 SEO
**核心问题**：做了独立站没人来访问全是白搭。

软件层面要做的：
- SSR / SSG（搜索引擎能爬）
- 完整 sitemap、robots.txt、Schema.org 结构化数据
- 页面速度优化
- 多语言 hreflang 标签
- Google Search Console / 百度站长接入

技术选型：Next.js / Nuxt / Astro 这类原生支持 SSR 的框架。

---

## B8. 支付与交易闭环
**核心问题**：用户付钱不是"调一个支付 API"那么简单。

核心抽象：
- 订单状态机：每个状态转换可追溯
- 支付幂等性：同一订单不能扣两次钱
- 异步回调可靠性：重复发送也安全

早期建议：不要自建支付，用 Shopify / 有赞 / Stripe Checkout。

---

## B9. 客户关系（CRM）
**核心问题**：用户从浏览 → 下载 App → 购买 → 售后，全链路数据应该连通。

核心抽象：
- CRM：用户统一档案（购买历史、设备绑定、客服记录、营销触点）
- CDP：进阶版，汇聚所有渠道数据

选项：腾讯企点、HubSpot、自建。早期 Excel 也比没有强。

---

## B10. 数据与分析
**核心问题**：你做了 5 个端，每个端的转化率、用户行为，看得见吗？

核心抽象：
- 埋点：用户行为被记录
- 数据仓库：所有端汇总到一处
- 看板：每天打开就能看的关键指标

工具：GA4（网站）、Firebase Analytics（App）、神策 / 友盟（国内）、Mixpanel。
关键概念：北极星指标——一个最重要的数字。

---

## B11. 触达系统
**核心问题**：用户买了产品就消失，怎么再次联系？

手段：
- 邮件：低成本、高度可定制（SendGrid、Mailchimp）
- App Push：免费、即时（FCM、个推）
- 短信：可达性最高但贵
- 公众号 + 模板消息：国内主战场
- 企业微信 / 加好友：高复购、高客单价产品的杀手锏

---

## B12. 客服与工单
**核心问题**：用户问题来自各种渠道，怎么统一管理？

核心抽象：
- 统一工单系统（Zendesk、Intercom、美洽、自建）
- 知识库（用户自助查）
- 客服后台（看到用户全部信息）

判断原则：客服压力 > 一周 5 小时时上工具。

---

## B13. 微信生态的特殊性
**国内市场必须重视**：

- 公众号：内容载体、品牌触达
- 小程序：轻量应用、不需下载
- 企业微信：客服、社群运营
- 视频号：直播带货
- 微信支付：支付通路
- 微信开放平台：账号互通

注意点：各产品独立 API、规则、审核流程；小程序大小限制；用户授权规则常变；政策风险。

---

## B14. 国际化
**核心决定**：要不要做海外？这决定今天的所有技术选型。

如果做海外：
- 架构层支持多语言、多币种、多区域部署
- 服务器多地域（CDN、多 region 数据库）
- 法务合规：GDPR
- 支付：Stripe / PayPal
- SEO：Google
- 触达：邮件为主

最忌讳"先做国内再说"——埋的雷三年后挖不出来。

---

## B15. 自动化与一人公司的极限
**核心策略**：把一切能自动化的都自动化，否则被运营琐事压垮。

需要自动化的：
- 代码部署：CI/CD（GitHub Actions / GitLab CI）
- 服务器运维：托管服务（Vercel、Railway、阿里云函数计算）
- 内容发布：CMS 直接发布
- 营销自动化：邮件序列、自动回复
- 财务对账：电商订单同步到记账软件
- 客户分层：基于行为自动打标签

原则：每件你重复做超过 3 次的事情，问"能不能自动化？"

---

## B16. 你自己的能力边界
**最难但最重要的一维**。

策略：
- **80/20 原则**：80% 功能用 60 分方案做出来（SaaS、模板、低代码）；20% 核心功能（固件、协议）用 95 分做
- **核心 vs 外包**：BLE 协议 / 固件 / 设备控制 App 必须自己掌握；CRM / 客服系统 / 邮件营销系统能用 SaaS 就用 SaaS
- **建立"我做什么 / 不做什么"清单**：写下来，每次有新想法对一遍
- **找一个能聊技术的同行**：付费咨询也比闭门造车强

---

# Part C：学习路径（按你的节奏推进）

抱着学习心态做这件事，需要一条由浅入深的路径。下面按 **优先级 + 学习曲线** 分四阶段。

## 第一阶段：把当前产品做扎实（0-3 个月）

围绕 RideWind 当前 stack（ESP32 + Flutter）。

| 学到的技术 | 对应维度 | AI 协作方式 |
|---|---|---|
| ESP32-IDF 工程化、组件化 | A1 代码可持续性 | 先让 AI 设计模块边界，你审查后再让 AI 实现 |
| BLE 协议设计（带版本号、扩展位） | A2 协议长期债务 | 让 AI 列出"未来 3 年可能要加的命令"，反推协议设计 |
| 固件日志框架 + Crashlytics 接入 | A3 可观测性 | 让 AI 写日志规范文档，再按规范实现 |
| ESP32 OTA 双分区 + 灰度策略 | A4 发布回滚 | OTA 是经典话题，AI 能直接给出方案 |
| 单元测试（Unity for C，flutter_test） | A8 可测试性 | 让 AI 帮你为每个新模块同时写测试 |
| ADR 决策记录文档 | A12 知识沉淀 | 每个重要决定让 AI 帮你写一份 ADR |

**关键里程碑**：第一版固件 + App 能稳定支撑 100 个真实用户。

---

## 第二阶段：扩展到全平台（3-9 个月）

| 学到的技术 | 对应维度 | AI 协作方式 |
|---|---|---|
| 后端开发（Node.js / Python + 数据库） | B3 账号、B4 数据 | 从一个最简 REST API 开始，逐步扩展 |
| 云服务部署（Vercel / Railway / 阿里云） | B15 自动化 | 用托管服务，避免运维深坑 |
| Next.js 官网 + CMS（Sanity / Strapi） | B6 内容系统、B7 SEO | AI 生成页面骨架你调样式 |
| 微信小程序（uni-app 或原生） | B1 平台、B13 微信 | 复用 App 的业务逻辑层 |
| Flutter Desktop（如果需要桌面端） | B1 平台 | 同一套 Flutter 代码改改就能用 |
| Firebase Analytics / 友盟接入 | B10 数据分析 | AI 帮你设计埋点方案 |
| 邮件 / Push 触达系统 | B11 触达 | 用 SaaS 接 API，不要自建 |
