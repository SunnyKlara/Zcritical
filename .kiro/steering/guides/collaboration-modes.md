# AI 协作模式手册

> 不同任务类型需要不同的协作节奏。选对模式 = 少走弯路。

---

## 模式 8：探索模式（Explore）

**适用场景：** 不确定方案、需要调研、技术选型、架构讨论

**节奏：**
1. AI 提出 2-3 个方案，每个方案列出 pros/cons
2. 用户选择方向或追问
3. 确认方案后切换到其他模式执行

**规则：**
- 不写代码，只输出分析和伪代码
- 每个方案必须有"最坏情况"分析
- 如果用户说"就这个"，立即切换到实现模式

**触发词：** "怎么做比较好"、"有什么方案"、"你觉得呢"、"对比一下"

**示例：**
```
用户：WiFi 音频投射延迟太高，怎么优化？
AI：三个方向——
  A) 减小 TCP buffer（降延迟但可能断续）
  B) 改 UDP（低延迟但丢包）
  C) 自适应 buffer（复杂但最优）
  各自 pros/cons...
```

## 模式 9：手术模式（Surgery）

**适用场景：** 修复已定位的 bug、精确改动已知位置的代码

**节奏：**
1. 确认病灶位置（文件 + 行号 + 根因）
2. 展示修复 diff（最小改动）
3. 验证编译通过
4. 更新相关文档（如果影响协议/架构）

**规则：**
- 只改必要的代码，不顺手重构
- 改动范围不超过 1-2 个文件
- 必须说明"为什么这样改能修复"
- 改完立即验证（build/analyze/test）

**触发词：** "这里有 bug"、"这行报错"、"修一下"、"FIX"

**示例：**
```
用户：ui_speed.c 第 85 行数组越界
AI：根因是 speed_index 没有 clamp 到 [0, MAX-1]。
    修复：加一行 if (idx >= MAX) idx = MAX - 1;
    [执行修改 + idf.py build 验证]
```

## 模式 10：脚手架模式（Scaffold）

**适用场景：** 新增功能模块、创建新文件、搭建框架

**节奏：**
1. 先回答"新功能三问"（属于哪层？放哪个文件？现有文件能容纳吗？）
2. 输出文件结构规划（不超过 3 个新文件）
3. 逐个文件实现，每个文件完成后编译验证
4. 最后补充文档（architecture.md 或 protocol-contract.md）

**规则：**
- 一次最多创建 3 个文件（教训 1）
- 新文件必须有头部注释说明职责
- 新增的公开 API 必须在创建时就确定签名，后续文件依赖此签名
- 如果涉及协议新增命令，先更新 protocol-contract.md 再写代码

**触发词：** "新增一个"、"加个功能"、"创建"、"搭建"

**示例：**
```
用户：加一个 OTA 升级的 UI 页面
AI：三问回答——
  1. 属于 ui/ 层
  2. 新建 ui_ota.c + ui_ota.h
  3. ui_manager.c 需要加一个 case
  [逐步实现]
```

## 模式 11：重构模式（Refactor）

**适用场景：** 拆分大文件、提取公共逻辑、改善代码结构

**节奏：**
1. 分析当前问题（为什么要重构？哪里违反了原则？）
2. 输出重构计划（拆成几步？每步改什么？）
3. 每步执行后验证：编译通过 + 行为不变
4. 全部完成后回归测试

**规则：**
- 重构不改行为——改完前后功能必须一致
- 每步都是可编译的中间状态（不允许"先全拆再组装"）
- 大文件拆分时，先提取最独立的部分（依赖最少的）
- 公开 API 不变（教训 7）

**触发词：** "拆一下"、"太大了"、"重构"、"提取"、"解耦"

**示例：**
```
用户：device_connect_screen.dart 太大了，拆一下 RGB 部分
AI：计划——
  Step 1: 提取 ColorizeRGBDetailView 到独立 widget 文件
  Step 2: 原文件 import 新 widget，删除旧代码
  Step 3: flutter analyze 验证
  [逐步执行]
```

## 模式切换

AI 应根据用户意图自动识别模式，但用户可以显式切换：

| 用户说 | 切换到 |
|--------|--------|
| "先讨论一下" | 探索模式 |
| "直接改" | 手术模式 |
| "新加一个 XX" | 脚手架模式 |
| "拆一下 / 重构" | 重构模式 |

混合场景：如果一个任务跨模式（比如"加个功能但要先讨论方案"），按顺序走：探索 → 确认 → 脚手架。
