# 开发工作流规范

## 核心原则

**不搞形式化流程。** 不创建 spec 文件（.kiro/specs/），不走 requirements → design → tasks 流水线。
设计讨论在对话中完成，决策记录在 CONTINUATION_GUIDE.md。

## 开发模式

```
1. 用户提需求（可能是一句话，可能是详细描述）
2. AI 读现有代码，理解当前架构、约束、已知坑位
3. AI 提出方案（技术选型、改动范围、影响分析、风险点）
4. 用户确认或调整方向
5. AI 直接写代码 + 编译验证（flutter analyze / idf.py build）
6. 用户烧录/运行实测，反馈结果
7. 迭代修复直到满意
8. commit + 更新 CONTINUATION_GUIDE.md
```

## 大功能开发方式

大功能（如车库大更新、灯光系统升级）不需要一次性设计完。按以下方式推进：

1. **开 feature 分支**（见 git-and-release.md 大功能分支规则）
2. **拆成小步骤**，每步是一个可验证的改动：
   - 先做数据结构/协议定义
   - 再做硬件端实现
   - 再做 APP 端 UI
   - 最后联调
3. **每步都是完整的**：能编译、能运行、不破坏现有功能
4. **每步结束 commit**，不攒大改动

## 新对话启动行为

每个新对话必须按顺序执行：
1. 读 `START-HERE.md` → 了解项目
2. 读 `CONTINUATION_GUIDE.md` → 了解当前状态和阻塞项
3. 读 `knowledge/known-pitfalls.md` → 了解踩过的坑（auto inclusion 会自动加载）
4. 读 `knowledge/lessons-learned.md` → 了解历史教训（auto inclusion 会自动加载）
5. `git status` → 确认工作区干净
6. **等用户说要做什么**，不要自作主张开始设计或创建文件

## 开始新功能前的强制 checklist

AI 在动手写任何代码之前，必须完成：
- [ ] 确认当前在正确的分支上（main 或对应 feature 分支）
- [ ] 确认工作区干净（git status = clean）
- [ ] 读了相关模块的现有代码（不是猜，是真的读了）
- [ ] 检查 known-pitfalls.md 中是否有相关坑位
- [ ] 提出方案并等用户确认
- [ ] 方案中明确列出要改哪些文件（不超过 3-5 个/步）

## 禁止事项

- ❌ 不要创建 `.kiro/specs/` 下的任何文件（除非用户明确说"创建 spec"）
- ❌ 不要在用户没确认方案前就开始写代码
- ❌ 不要一次性改 >5 个文件而不先说明改动计划
- ❌ 不要引入用户没提到的新框架/库/架构模式

## 方案讨论格式

当 AI 提出方案时，应包含：

```
## 方案：[一句话标题]

**改什么**：列出要改的文件和大致改动
**怎么改**：技术方案简述
**影响**：对现有功能的影响、是否需要新协议命令
**风险**：可能的坑、硬件限制、兼容性问题
**步骤**：拆成 2-5 步，每步可独立验证
```

用户说"可以"后才动手。

## 对话结束时的持久化

除了更新 CONTINUATION_GUIDE.md，还需要检查：

1. **设计决策** — 如果本次对话确定了某个功能的方向、风格、交互方式，追加到 `product-vision.md` 的对应段落
2. **否决方案** — 如果讨论中否决了某个方案，追加到 `product-vision.md` 的"否决过的方案"列表
3. **新坑位** — 如果开发中踩了新坑，追加到 `known-pitfalls.md`
4. **新教训** — 如果犯了流程/协作错误，追加到 `lessons-learned.md`

目的：让下一个对话的 AI 不需要重新讨论已经决定过的事情。
