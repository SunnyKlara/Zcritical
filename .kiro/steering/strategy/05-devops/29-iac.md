# 29. 基础设施即代码（IaC）

## 是什么
用代码定义和管理基础设施（服务器、数据库、网络、DNS）。不再手动在云控制台点点点，而是写配置文件，`terraform apply` 一键创建/更新/销毁。

## 为什么需要
- 手动创建资源容易遗漏/出错（忘记开端口、安全组配错）
- 环境一致性：开发/测试/生产用同一份代码创建
- 可审计：基础设施变更有 git 历史
- 可复现：灾难恢复时一键重建整套环境
- 团队协作：新人看代码就知道架构，不用问"这台机器是干嘛的"

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              IaC 架构                                 │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [代码仓库] infra/                                  │
│   ├─ main.tf          (主配置)                      │
│   ├─ variables.tf     (变量定义)                    │
│   ├─ outputs.tf       (输出值)                      │
│   ├─ terraform.tfvars (环境变量)                    │
│   └─ modules/                                       │
│       ├─ vpc/         (网络)                        │
│       ├─ ecs/         (服务器)                      │
│       ├─ rds/         (数据库)                      │
│       └─ oss/         (存储)                        │
│                                                      │
│  [工作流]                                            │
│   terraform plan → 预览变更                         │
│   terraform apply → 执行变更                        │
│   terraform destroy → 销毁环境                      │
│                                                      │
│  [状态管理]                                          │
│   terraform.tfstate → 远程存储（OSS/S3）           │
│   状态锁：防止并发修改                             │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| IaC 工具 | Terraform | 多云支持，社区最大 |
| 备选 | Pulumi (代码式) / CloudFormation (AWS) | 各有优势 |
| Provider | alicloud / aws | 阿里云/AWS 资源管理 |
| 状态存储 | 阿里云 OSS + TableStore | 远程状态 + 锁 |
| 密钥管理 | 环境变量 / Vault | AK/SK 不入代码 |
| CI 集成 | GitHub Actions + terraform | PR 时 plan，merge 时 apply |

## 实现步骤

### Phase 1：基础环境（2h）
1. Terraform 安装 + 阿里云 Provider 配置
2. 远程状态存储配置（OSS backend）
3. 创建第一个资源：VPC + 安全组

### Phase 2：核心资源（2-3h）
4. ECS 实例（应用服务器）
5. RDS PostgreSQL（数据库）
6. Redis 实例
7. OSS Bucket（文件存储）
8. 域名 + SSL 证书

### Phase 3：模块化（1-2h）
9. 抽取可复用模块（VPC/ECS/RDS）
10. 环境分离：dev/staging/prod 用不同 tfvars
11. 输出关键信息（IP、连接字符串、域名）

### Phase 4：CI 集成（1-2h）
12. PR 时自动 `terraform plan`（预览变更）
13. 合并后自动 `terraform apply`（执行变更）
14. 销毁保护：生产环境禁止 destroy

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 状态文件丢失 | 无法管理已有资源 | 远程存储 + 备份 |
| 手动修改资源 | 状态和实际不一致 | 禁止手动改，全部走代码 |
| 密钥泄露 | AK/SK 在代码中 | 环境变量 + .gitignore |
| destroy 误操作 | 生产数据库被删 | 生命周期 prevent_destroy |
| 依赖顺序 | 资源创建顺序错误 | Terraform 自动处理，显式 depends_on |

## 与 RideWind 的关系
- 当前状态：无云资源
- 优先级：P4（后端上云时同步做）
- 起步：一份 Terraform 代码管理所有阿里云资源
- 价值：换台服务器/灾难恢复时一键重建
