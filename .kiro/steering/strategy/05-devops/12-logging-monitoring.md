# 12. 日志收集与监控

## 是什么
集中收集所有服务的日志，实时监控系统指标（CPU/内存/QPS/错误率），异常时自动告警。让你"知道系统现在怎么样"而不是"用户告诉你挂了"。

## 为什么需要
- 没有监控 = 盲人开车，出了问题不知道原因
- 分布式系统日志分散在多台机器，需要集中查看
- 性能问题需要指标趋势分析（什么时候开始慢的？）
- 告警：CPU>90% / 错误率>5% / 磁盘快满 → 立即通知
- 事后分析：出了故障能快速定位根因

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              监控体系                                 │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [日志]                                              │
│   应用日志 → Filebeat/Fluentd → Elasticsearch       │
│   → Kibana 可视化查询                               │
│   轻量替代：Loki + Grafana                          │
│                                                      │
│  [指标]                                              │
│   Prometheus 拉取 → 时序存储                        │
│   → Grafana 看板（CPU/内存/QPS/延迟）              │
│                                                      │
│  [告警]                                              │
│   Prometheus Alertmanager / Grafana Alerts           │
│   → 飞书/钉钉/邮件/短信                            │
│                                                      │
│  [追踪]                                              │
│   Jaeger / Zipkin                                   │
│   → 跨服务调用链可视化                             │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 日志 | Loki + Grafana | 轻量，和 Prometheus 生态统一 |
| 指标 | Prometheus + Grafana | 业界标准，免费 |
| 告警 | Grafana Alerting | 统一告警管理 |
| 追踪 | Jaeger | 分布式链路追踪 |
| 云托管 | 阿里云 SLS / AWS CloudWatch | 零运维 |
| 应用埋点 | prom-client (Node) / prometheus (Go) | 自定义指标 |

## 实现步骤

### Phase 1：日志规范（1h）
1. 统一日志格式：JSON（timestamp, level, service, message, trace_id）
2. 日志级别规范：ERROR（需要处理）/ WARN（需要关注）/ INFO（正常流程）/ DEBUG
3. 敏感信息脱敏（密码、token 不打印）

### Phase 2：指标采集（2-3h）
4. Prometheus 部署（Docker）
5. 应用暴露 /metrics 端点
6. 关键指标：HTTP QPS、延迟 P99、错误率、活跃连接数
7. 系统指标：node_exporter（CPU/内存/磁盘/网络）

### Phase 3：可视化看板（2h）
8. Grafana 部署 + 数据源配置
9. 预置看板：系统概览、API 性能、数据库、Redis
10. 自定义看板：业务指标（日活、OTA 成功率）

### Phase 4：告警配置（1-2h）
11. 告警规则：CPU>80% 持续 5min / 错误率>5% / 磁盘>85%
12. 告警通道：飞书 Webhook / 邮件
13. 告警分级：P0（立即处理）/ P1（1小时内）/ P2（工作时间）
14. 静默规则：维护窗口不告警

### Phase 5：日志收集（2h）
15. Loki 部署 + Promtail 采集容器日志
16. Grafana 中查询日志（按服务/级别/关键词过滤）
17. 日志保留策略：7天热存储，30天冷存储

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 日志量爆炸 | 存储成本高，查询慢 | 采样 + 级别控制 + 保留策略 |
| 告警疲劳 | 太多告警导致忽略 | 合理阈值 + 分级 + 聚合 |
| 指标基数爆炸 | Prometheus OOM | 避免高基数 label（如 user_id） |
| 时钟不同步 | 日志时间对不上 | 所有机器 NTP 同步 |
| 监控系统本身挂了 | 无法发现问题 | 监控系统自身的健康检查 |
