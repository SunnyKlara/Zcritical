# 13. 自动化测试体系

## 是什么
多层次的自动化测试：单元测试、集成测试、端到端测试、性能测试。覆盖固件、App、后端三端。让每次修改都有信心不会破坏已有功能。

## 为什么需要
- 没有测试 = 每次修改都是赌博
- 回归测试手动做太慢（改一行代码要测一小时）
- CI 门禁需要自动化测试支撑
- 重构的前提：有测试才敢大改
- 团队协作：别人的代码改了你的功能，测试能发现

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              测试金字塔                               │
├─────────────────────────────────────────────────────┤
│                                                      │
│         /\        E2E 测试（少量）                   │
│        /  \       真机/模拟器完整流程               │
│       /    \                                         │
│      /──────\     集成测试（适量）                   │
│     / 多模块 \    API 测试 / BLE 协议测试           │
│    /  协作验证 \                                     │
│   /────────────\  单元测试（大量）                   │
│  / 函数级别验证 \ 纯逻辑，快速，无依赖             │
│ /________________\                                   │
│                                                      │
│  [ESP32] Unity 框架 / 桩函数模拟硬件               │
│  [Flutter] flutter test / integration_test          │
│  [后端] Jest / Go testing / Supertest               │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 端 | 框架 | 说明 |
|----|------|------|
| ESP32 单元测试 | Unity (C 测试框架) | 轻量，适合嵌入式 |
| ESP32 模拟 | CMock / 手写桩函数 | 模拟硬件 API |
| Flutter 单元 | flutter test + mockito | Widget 测试 + 逻辑测试 |
| Flutter 集成 | integration_test | 真机/模拟器 UI 测试 |
| 后端单元 | Jest (Node) / testing (Go) | 标准选择 |
| 后端 API | Supertest / httptest | HTTP 接口测试 |
| 覆盖率 | lcov / Codecov | 覆盖率报告和趋势 |

## 实现步骤

### Phase 1：Flutter 测试（2-3h）
1. 协议解析单元测试（已有 51 个，继续扩展）
2. Provider 逻辑测试（mock BLE service）
3. Widget 测试（关键页面渲染验证）
4. 覆盖率目标：核心逻辑 >80%

### Phase 2：ESP32 测试（2-3h）
5. 协议解析函数测试（纯 C 逻辑，可在 PC 上跑）
6. 状态机测试（app_state 状态转换）
7. 音频混合算法测试（输入→输出验证）
8. 桩函数：模拟 gpio_get_level / esp_timer_get_time 等

### Phase 3：后端测试（2h）
9. API 接口测试（注册/登录/设备管理）
10. 数据库操作测试（事务/约束/边界）
11. 测试数据库：每次测试前清空，用 fixture 填充

### Phase 4：集成测试（2-3h）
12. BLE 协议端到端：模拟 App 发送命令 → 验证设备响应
13. OTA 流程测试：上传固件 → 灰度规则 → 设备拉取
14. 用 Docker Compose 启动完整环境跑集成测试

### Phase 5：CI 集成（1h）
15. PR 必须通过所有测试
16. 覆盖率报告自动评论到 PR
17. 测试失败自动通知

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 测试太慢 | 开发者不愿意跑 | 分层：单元测试秒级，集成测试分钟级 |
| 测试不稳定 | CI 随机失败 | 消除外部依赖，用 mock |
| 过度 mock | 测试通过但实际有 bug | 适度集成测试验证真实交互 |
| 维护成本 | 改代码要改测试 | 测试行为不测实现 |
| 硬件依赖 | ESP32 测试需要真机 | 纯逻辑抽离到可 PC 测试的模块 |
