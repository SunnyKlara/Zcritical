# 30. 服务网格（Service Mesh）

## 是什么
在微服务之间插入一层透明代理（Sidecar），统一处理服务间通信的复杂性：负载均衡、熔断、重试、mTLS 加密、流量观测。应用代码不需要关心这些。

## 为什么需要
- 微服务间通信逻辑（重试/熔断/超时）不应该写在业务代码中
- 统一的 mTLS：服务间通信自动加密，零信任网络
- 流量管理：金丝雀发布、流量镜像、故障注入
- 可观测性：自动采集服务间调用指标，无需改代码
- 注意：这是高级话题，小规模不需要

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              Service Mesh 架构                        │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [控制平面] Istio / Linkerd                         │
│   配置下发 / 证书管理 / 策略定义                   │
│       │                                              │
│       ▼                                              │
│  [数据平面] Envoy Sidecar (每个 Pod 一个)           │
│                                                      │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐        │
│  │ Service A│    │ Service B│    │ Service C│        │
│  │ ┌─────┐ │    │ ┌─────┐ │    │ ┌─────┐ │        │
│  │ │ App │ │    │ │ App │ │    │ │ App │ │        │
│  │ └──┬──┘ │    │ └──┬──┘ │    │ └──┬──┘ │        │
│  │ ┌──▼──┐ │    │ ┌──▼──┐ │    │ ┌──▼──┐ │        │
│  │ │Envoy│◄├────┤►│Envoy│◄├────┤►│Envoy│ │        │
│  │ └─────┘ │    │ └─────┘ │    │ └─────┘ │        │
│  └─────────┘    └─────────┘    └─────────┘        │
│                                                      │
│  Envoy 自动处理：                                   │
│   mTLS / 负载均衡 / 重试 / 熔断 / 指标采集        │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 控制平面 | Istio / Linkerd | Istio 功能全，Linkerd 更轻量 |
| 数据平面 | Envoy | 高性能 L7 代理 |
| 平台 | Kubernetes | Service Mesh 依赖 K8s |
| 可观测 | Kiali (Istio 可视化) | 服务拓扑图 |
| 流量管理 | VirtualService / DestinationRule | Istio CRD |

## 实现步骤

### Phase 1：理解概念（2h）
1. Sidecar 模式原理
2. 控制平面 vs 数据平面
3. 什么时候需要 Service Mesh（>10 个微服务时考虑）

### Phase 2：Istio 安装（2h）
4. K8s 集群上安装 Istio（istioctl install）
5. 命名空间注入 Sidecar（label: istio-injection=enabled）
6. 验证：部署示例应用，观察 Envoy 自动注入

### Phase 3：流量管理（2-3h）
7. 金丝雀发布：90% 流量到 v1，10% 到 v2
8. 故障注入：模拟 Service B 延迟 5s，测试 A 的熔断
9. 流量镜像：生产流量复制到测试环境

### Phase 4：安全（1-2h）
10. mTLS：服务间通信自动加密
11. 授权策略：Service A 只能调用 Service B 的特定接口
12. 证书自动轮换

### Phase 5：可观测性（1h）
13. Kiali：服务拓扑图 + 流量可视化
14. 自动指标：请求量/延迟/错误率（无需改代码）
15. 分布式追踪集成

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 过早引入 | 复杂度远超收益 | <10 个服务不需要 |
| 性能开销 | 每次调用多一跳（Envoy） | 延迟增加 1-3ms，大多数场景可接受 |
| 调试困难 | 问题可能在 Envoy 层 | 学会看 Envoy 日志和指标 |
| 资源消耗 | 每个 Pod 多一个 Envoy 容器 | 内存 +50-100MB/Pod |
| 学习曲线 | Istio 概念多，配置复杂 | 从 Linkerd 入门（更简单） |
