# 43. 多设备协同

## 是什么
一个用户拥有多台 RideWind 设备，App 统一管理，设备间可以联动/同步。包括设备列表管理、快速切换、分组控制、灯效同步（主从模式）、配置克隆。

## 为什么需要
- 用户可能有多辆车/多个场景：公路车+山地车各装一台
- 设备间灯效同步很酷：车队骑行时所有设备同步闪烁
- 提升客单价：一个用户买多台设备
- 配置复用：调好一台的配置，一键克隆到另一台
- 未来扩展：前灯+尾灯联动、多设备组网

## 技术架构
```
┌─────────────────────────────────────────────────────┐
│              多设备协同架构                           │
├─────────────────────────────────────────────────────┤
│                                                      │
│  [App 端]                                            │
│   设备列表管理 → 快速切换 → 分组控制              │
│       │                                              │
│       ├─ [单设备模式] 当前只连接一台               │
│       │   切换设备：断开A → 连接B（~2秒）         │
│       │                                              │
│       ├─ [多设备模式] 同时连接多台                  │
│       │   BLE 多连接（iOS最多7台，Android最多5台） │
│       │   统一控制面板 → 广播命令到所有设备        │
│       │                                              │
│       └─ [同步模式] 设备间灯效同步                 │
│           App 作为中心节点 → 同时发送相同命令      │
│           或 设备间 BLE Mesh（高级）               │
│       │                                              │
│       ▼                                              │
│  [设备端]                                            │
│   接收命令 → 执行 → 上报状态                      │
│   每台设备独立运行，不依赖其他设备                 │
│                                                      │
└─────────────────────────────────────────────────────┘
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| BLE 多连接 | flutter_reactive_ble 多实例 | 同时维护多个连接 |
| 设备管理 | 本地数据库 (SQLite/Hive) | 设备列表持久化 |
| 同步协议 | App 广播模式 | App 同时向多台设备发送相同命令 |
| 高级同步 | BLE Mesh (ESP-BLE-MESH) | 设备间直接通信（未来） |
| 配置克隆 | 读取A配置 → 写入B | 通过 App 中转 |
| 分组 | App 本地分组逻辑 | 用户自定义设备分组 |

## 实现步骤

### Phase 1：设备列表管理（2h）
1. 设备数据模型：
   ```dart
   class DeviceInfo {
     final String sn;
     final String name;
     final String? group;
     final String macAddress;
     final String firmwareVer;
     final DateTime lastSeen;
     final Map<String, dynamic> lastConfig;
   }
   ```
2. 设备发现与添加：扫描 BLE → 识别 RideWind → 添加到列表
3. 设备管理操作：重命名/分组/删除/排序

### Phase 2：快速切换（2h）
4. 切换流程优化：断开A → 连接B（~1-2秒）→ 加载B配置到UI
5. 最近使用排序：最近连接的设备排在列表顶部
6. 自动重连：App 启动时自动连接上次使用的设备
7. 连接状态指示：列表中显示每台设备的在线/离线状态

### Phase 3：分组控制（2h）
8. 分组功能：创建分组 → 将设备拖入 → 对分组执行操作
9. 组控制命令：并行发送命令到组内所有设备
10. 组状态展示：组内设备状态汇总

### Phase 4：灯效同步（2-3h）
11. 同步模式设计：
    - 主从模式：主设备变化 → App 监听 → 同步到从设备
    - 广播模式：App 统一控制面板 → 同时发送到所有设备
12. 同步精度：BLE 延迟 ~10-50ms，多设备间差 ~20-100ms
13. 时间戳同步：命令附带时间戳，设备在指定时间开始播放

### Phase 5：配置克隆（1h）
14. 克隆流程：选择源设备A → 读取全部配置 → 写入目标设备B
15. 选择性克隆：只克隆部分配置（灯效/音效/通用）
16. 配置模板：保存配置为模板 → 应用到任意设备

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| BLE 多连接不稳定 | 连接数多时频繁断连 | 限制同时连接数（建议≤3） |
| iOS 后台限制 | App 进后台后断开 | Background Mode + 心跳保活 |
| 同步延迟 | 多台设备灯效不同步 | 时间戳对齐 + 容忍 50ms 误差 |
| 设备识别混淆 | 多台设备 MAC 相似 | 用 SN 区分 + 用户自定义名称 |
| 电量消耗 | 多连接增加手机功耗 | 非活跃设备降低连接间隔 |
| Android 碎片化 | 不同手机最大连接数不同 | 运行时检测 + 优雅降级 |

## 与 RideWind 的关系
- 当前状态：App 只支持单设备连接，无设备列表管理
- 优先级：P4（产品稳定 + 有多设备用户后）
- 起步方案：设备列表 + 快速切换（不需要同时连接）
- 进阶：多设备同时连接 + 灯效同步
- 场景：车队骑行同步灯效、前后灯联动、多车管理
