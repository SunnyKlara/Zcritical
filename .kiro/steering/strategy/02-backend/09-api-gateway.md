# 9. API 网关

## 是什么
所有客户端请求的统一入口。负责路由转发、认证鉴权、限流熔断、日志记录、协议转换。是微服务架构的"门卫"。

## 为什么需要
- 客户端只需知道一个地址，不需要知道后端有多少个服务
- 统一认证：JWT 验证在网关做一次，后端服务不重复做
- 统一限流：防止恶意请求打垮后端
- 统一日志：所有请求的 access log 集中记录
- API 版本管理：v1/v2 路由到不同服务

## 技术栈选型
| 组件 | 技术 | 理由 |
|------|------|------|
| 轻量方案 | Nginx + Lua (OpenResty) | 性能极高，配置简单 |
| 中量方案 | Kong / APISIX | 插件生态丰富，管理界面 |
| 云原生 | 阿里云 API Gateway | 零运维，按调用付费 |
| 自研 | Node.js / Go 反向代理 | 完全可控，学习价值高 |
| SSL | Let's Encrypt + Certbot | 免费自动续期 |

## 实现步骤

### Phase 1：基础路由（1-2h）
1. Nginx 反向代理配置：`/api/user/*` → user-service:3001
2. SSL 证书配置（Let's Encrypt）
3. CORS 头统一设置

### Phase 2：认证中间件（2h）
4. JWT 验证：从 Authorization header 提取 token → 验证签名 → 注入 user_id 到请求头
5. 白名单路由：注册/登录/公开接口跳过认证
6. 角色鉴权：admin 路由需要 role=admin

### Phase 3：限流与熔断（2h）
7. 限流策略：令牌桶算法，按 IP / 用户 / API 分别限制
8. 熔断：后端服务 5xx 超过阈值 → 快速失败返回 503
9. 降级：非核心接口熔断时返回缓存数据

### Phase 4：日志与监控（1-2h）
10. Access Log：请求时间、路径、状态码、耗时、用户ID
11. 慢请求告警：响应 >2s 的请求记录并报警
12. 流量统计：QPS、错误率、P99 延迟

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 单点故障 | 网关挂了全站不可用 | 多实例 + 负载均衡 |
| 性能瓶颈 | 所有流量经过网关 | Nginx 性能够用，或水平扩展 |
| 配置错误 | 路由错误导致 404/502 | 配置版本管理 + 灰度发布 |
| JWT 时钟偏差 | 不同服务器时间不同步 | NTP 同步 + 允许 30s 偏差 |
| 大文件上传 | 网关超时/内存溢出 | 直传 OSS，不经过网关 |

## 与 RideWind 的关系
- 当前状态：无后端
- 优先级：P3（后端搭建时一起做）
- 起步方案：单台 Nginx 反向代理（够用到万级用户）
- 演进：用户量上来后换 Kong 或阿里云 API Gateway
