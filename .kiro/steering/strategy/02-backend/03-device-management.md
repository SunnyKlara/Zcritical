# 3. 后端设备管理

## 是什么

管理所有出厂设备的生命周期：注册、激活、状态监控、固件版本追踪、远程配置下发。是 OTA、远程诊断、数据分析的基础设施。

## 为什么需要

- 知道"世界上有多少台我的设备在运行"
- OTA 灰度需要按设备属性筛选（版本、地区、批次）
- 远程诊断需要查询设备状态
- 售后需要通过 SN 查到设备信息
- 产品决策需要设备活跃数据

## 技术架构

```
┌──────────────────────────────────────────────────┐
│               设备管理服务                         │
├──────────────────────────────────────────────────┤
│                                                   │
│  [设备注册]                                       │
│   产线烧录时写入 SN → 首次联网时上报注册         │
│                                                   │
│  [设备状态]                                       │
│   在线/离线 / 固件版本 / 最后心跳时间            │
│   硬件版本 / 绑定用户 / 激活时间                 │
│                                                   │
│  [远程配置]                                       │
│   下发参数（默认亮度、默认模式、功能开关）       │
│   设备下次连接时拉取最新配置                     │
│                                                   │
│  [批量操作]                                       │
│   按条件筛选 → 批量推送 OTA / 配置 / 通知       │
│                                                   │
└──────────────────────────────────────────────────┘
```

## 技术栈选型

| 组件 | 技术 | 说明 |
|------|------|------|
| 数据库 | PostgreSQL | 设备表、状态表、配置表 |
| 缓存 | Redis | 在线状态、心跳 TTL |
| 设备标识 | SN (eFuse) + MAC 地址 | 双重标识防伪造 |
| API | RESTful + WebSocket | 管理后台用 REST，实时状态用 WS |
| 消息 | MQTT / HTTP 长轮询 | 设备拉取配置 |

## 实现步骤

### Phase 1：设备数据模型（1-2h）

1. **核心表设计**
   ```sql
   CREATE TABLE devices (
       sn              VARCHAR(50) PRIMARY KEY,
       mac_address     VARCHAR(17),
       hardware_ver    VARCHAR(20),
       firmware_ver    VARCHAR(20),
       owner_id        UUID REFERENCES users(id),
       activated_at    TIMESTAMPTZ,
       last_seen       TIMESTAMPTZ,
       status          VARCHAR(20) DEFAULT 'inactive',
       region          VARCHAR(50),
       batch_id        VARCHAR(50),
       created_at      TIMESTAMPTZ DEFAULT NOW()
   );
   
   CREATE TABLE device_configs (
       sn              VARCHAR(50) REFERENCES devices(sn),
       config_key      VARCHAR(100),
       config_value    JSONB,
       updated_at      TIMESTAMPTZ DEFAULT NOW(),
       PRIMARY KEY (sn, config_key)
   );
   ```

### Phase 2：设备注册与激活（2h）

2. **注册流程**
   - 设备首次通过 App 联网 → App 调用注册 API
   - 携带 SN + MAC + 硬件版本 + 固件版本
   - 后端验证 SN 合法性 → 创建设备记录

3. **心跳机制**
   - App 连接设备期间定期上报（每 5 分钟）
   - Redis TTL key 管理在线状态
   - 超时未心跳 → 自动标记 offline

### Phase 3：管理 API（2-3h）

4. **设备列表与筛选**
   - 分页、排序、多条件筛选（状态/版本/批次/地区）
   - 支持导出 CSV

5. **远程配置下发**
   - 管理员设置配置 → 设备下次连接时 App 拉取 → BLE 下发
   - 明确优先级：用户手动设置 > 云端默认配置

6. **批量操作**
   - 按条件选择设备集合 → 批量推送 OTA / 修改配置
   - 异步执行，返回任务 ID，可查询进度

### Phase 4：统计与看板（1-2h）

7. **关键指标**
   - 总设备数 / 激活数 / 日活 / 月活
   - 固件版本分布 / 地区分布
   - 批次良率关联

## 关键坑点

| 坑 | 后果 | 解法 |
|----|------|------|
| SN 伪造 | 假设备注册占用资源 | SN 算法校验 + 产线预注册白名单 |
| 心跳风暴 | 万台设备同时心跳压垮服务器 | 随机抖动 + Redis 原子操作 |
| 配置冲突 | 用户本地设置 vs 云端配置 | 明确优先级规则 |
| 设备转让 | 旧用户数据残留 | 解绑时清除关联 |
| 离线设备 | 配置下发不到 | 存储待下发队列，上线时同步 |
