# 25. 工单/客服系统

## 是什么
用户提交问题→客服处理→跟踪→解决→满意度评价的完整闭环。是产品售后服务的核心基础设施，让用户有渠道反馈问题，让团队有流程化的方式处理问题，同时积累问题数据用于产品改进。

## 为什么需要
- 用户遇到问题需要有渠道反馈（不能只靠微信群）
- 售后问题需要流程化管理：谁负责、什么时候解决、是否超时
- 问题数据可分析改进产品：Top 10 问题 → 优先修复
- SLA 管理：承诺 24h 内响应，48h 内解决
- 用户满意度追踪：解决后评价，量化服务质量

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 自建方案 | Node.js + PostgreSQL | 完全可控，定制性强 |
| 第三方 | Zendesk / Freshdesk | 开箱即用，但月费较高 |
| 轻量方案 | 飞书多维表格 + 机器人 | 零成本起步 |
| 状态机 | XState / 自研 FSM | 工单生命周期管理 |
| 通知 | 微信模板消息 + App 推送 | 状态变更实时通知用户 |
| 知识库 | Markdown + 搜索 | FAQ 自助解决 |
| SLA 引擎 | Cron + Redis TTL | 超时自动升级 |

## 实现步骤

### Phase 1：工单数据模型（2h）
1. 核心表设计：
   ```sql
   CREATE TABLE tickets (
     id            UUID PRIMARY KEY,
     user_id       UUID REFERENCES users(id),
     device_sn     VARCHAR(50),
     category      VARCHAR(50),    -- hardware/software/usage/other
     priority      SMALLINT,       -- 1=紧急 2=高 3=中 4=低
     title         VARCHAR(200),
     description   TEXT,
     attachments   JSONB,          -- ["url1", "url2"]
     status        VARCHAR(20),    -- open/assigned/in_progress/pending_user/closed
     assignee_id   UUID,
     sla_deadline  TIMESTAMPTZ,
     resolved_at   TIMESTAMPTZ,
     satisfaction  SMALLINT,       -- 1-5 星
     created_at    TIMESTAMPTZ DEFAULT NOW(),
     updated_at    TIMESTAMPTZ DEFAULT NOW()
   );

   CREATE TABLE ticket_messages (
     id            UUID PRIMARY KEY,
     ticket_id     UUID REFERENCES tickets(id),
     sender_type   VARCHAR(10),    -- user/agent/system
     sender_id     UUID,
     content       TEXT,
     attachments   JSONB,
     created_at    TIMESTAMPTZ DEFAULT NOW()
   );
   ```
2. 分类标签体系：硬件故障/软件Bug/使用咨询/功能建议/退换货
3. 优先级规则：设备无法使用=紧急，功能异常=高，使用问题=中，建议=低

### Phase 2：工单生命周期管理（2-3h）
4. 状态机定义：
   ```
   open → assigned（自动/手动分配）
   assigned → in_progress（客服开始处理）
   in_progress → pending_user（等待用户补充信息）
   pending_user → in_progress（用户回复）
   in_progress → closed（问题解决）
   任何状态 → escalated（SLA 超时升级）
   ```
5. SLA 规则引擎：
   - 紧急：1h 响应，4h 解决
   - 高：4h 响应，24h 解决
   - 中：24h 响应，72h 解决
   - 低：48h 响应，7天解决
6. 超时处理：Redis TTL 到期 → 自动升级优先级 + 通知主管

### Phase 3：客服工作台（3h）
7. 工单列表：按优先级/状态/时间排序，支持筛选
8. 工单详情：用户信息 + 设备信息 + 对话历史 + 内部备注
9. 快捷回复模板：常见问题预设回复（"请尝试重启设备..."）
10. 批量操作：批量关闭/分配/标记

### Phase 4：用户端体验（2h）
11. App 内提交入口：选分类 → 填描述 → 上传截图/日志
12. 自动附带设备信息：SN、固件版本、App 版本、手机型号
13. 工单进度查看：状态时间线 + 客服回复
14. 满意度评价：关闭后 24h 推送评价请求

### Phase 5：数据分析（1h）
15. 核心指标看板：
    - 工单量趋势（日/周/月）
    - 平均响应时间 / 平均解决时间
    - SLA 达标率 / 满意度评分分布
    - Top 10 问题分类
16. 问题闭环：高频问题 → 产品改进任务 → 验证效果

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 分类不合理 | 用户选错分类，客服分配错误 | 支持客服重新分类 + AI 辅助分类 |
| SLA 计算复杂 | 非工作时间是否算入 | 明确定义：只算工作时间 or 自然时间 |
| 重复工单 | 同一问题多次提交 | 检测相似工单 + 合并功能 |
| 用户不回复 | 工单一直挂着 | 7天无回复自动关闭 + 提醒 |
| 客服离职 | 工单无人处理 | 自动重新分配 + 主管兜底 |
| 恶意投诉 | 浪费客服资源 | 用户信用体系 + 频率限制 |

## 与 RideWind 的关系
- 当前状态：无客服系统，用户问题通过微信群人工处理
- 优先级：P5（用户量到百级后）
- 起步方案：飞书多维表格 + 机器人（零成本）→ 自建系统
- 依赖：#2 用户系统（关联用户）+ #3 设备管理（关联设备）
- 过渡：微信群 → 飞书表格 → 自建工单系统
