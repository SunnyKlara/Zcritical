# 50. 售后数据闭环

## 是什么
将售后问题数据反馈到产品改进流程，形成"问题→分析→改进→验证"的闭环。不只是解决单个用户的问题，而是从问题数据中发现系统性缺陷，推动产品/固件/文档的持续改进，让同样的问题不再反复出现。

## 为什么需要
- 同样的问题不应该反复出现：第 100 个用户问同样的问题 = 产品缺陷
- 售后数据是产品改进的金矿：用户用脚投票，问题数据最真实
- 量化产品质量：故障率、返修率、满意度都是可追踪的指标
- 闭环验证：改进后问题是否真的减少了？数据说了算
- 降低售后成本：根治问题 > 反复处理同类工单

## 技术架构
```

│   │  关联分析 → 固件版本/批次/使用场景       │       │
```

## 技术栈选型
| 组件 | 技术 | 说明 |
|------|------|------|
| 数据源 | 工单系统 + 诊断系统 | 结构化问题数据 |
| 分析工具 | Metabase / SQL | 自助分析 + 看板 |
| 分类体系 | 标签系统 (PostgreSQL JSONB) | 多维度标记 |
| 任务追踪 | 飞书项目 / GitHub Issues | 改进任务管理 |
| 报告生成 | 自动化脚本 + 模板 | 周报/月报自动生成 |
| 关联分析 | SQL JOIN + 统计 | 问题与版本/批次关联 |

## 实现步骤

### Phase 1：问题分类体系（1-2h）
1. 建立多级分类：
   ```
   一级分类：硬件 / 软件 / 使用 / 物流
   二级分类：
     硬件 → LCD/LED/喇叭/编码器/电源/散热
     软件 → 固件崩溃/BLE连接/音频异常/显示异常
     使用 → 操作不会/功能误解/配置问题
     物流 → 包装损坏/配件缺失/发错货
   ```
2. 标签系统：每个工单可打多个标签
   - 固件版本标签：v1.0.0 / v1.1.0 / v1.2.0
   - 批次标签：batch_2025Q1 / batch_2025Q2
   - 严重度标签：critical / major / minor / cosmetic
3. 自动标签：根据错误码自动关联分类

### Phase 2：Top N 问题看板（2h）
4. 核心看板设计：
   ```sql
   -- Top 10 问题（按工单量排序）
   SELECT category, subcategory, count(*) as ticket_count,
          avg(resolution_hours) as avg_resolve_time
   FROM tickets
   WHERE created_at > now() - interval '30 days'
   GROUP BY category, subcategory
   ORDER BY ticket_count DESC
   LIMIT 10;
   ```
5. 趋势对比：本周 vs 上周 vs 上月，问题量变化
6. 关联分析：
   ```sql
   -- 某固件版本的问题分布
   SELECT firmware_version, category, count(*)
   FROM tickets t JOIN devices d ON t.device_sn = d.sn
   GROUP BY firmware_version, category;
   ```
7. 批次质量追踪：不同生产批次的故障率对比

### Phase 3：改进任务追踪（2h）
8. 从问题到任务的转化流程：
   ```
   Top N 问题 → 根因分析 → 改进方案 → 创建任务
   任务属性：
     - 关联问题分类
     - 负责人
     - 预计完成时间
     - 改进类型（固件/硬件/文档/流程）
     - 验证指标（改进后问题量目标）
   ```
9. 改进任务看板：待处理 / 进行中 / 已完成 / 已验证
10. 定期评审：每周回顾 Top N 问题 + 改进进度

### Phase 4：效果验证（1-2h）
11. 验证指标定义：
    - 改进前：过去 30 天该类问题工单数
    - 改进后：发布修复后 30 天该类问题工单数
    - 目标：下降 >50%
12. 自动验证报告：
    ```sql
    SELECT 
      count(*) FILTER (WHERE created_at < fix_date) as before_count,
      count(*) FILTER (WHERE created_at >= fix_date) as after_count
    FROM tickets
    WHERE category = 'ble_disconnect'
      AND created_at BETWEEN fix_date - interval '30 days' 
                         AND fix_date + interval '30 days';
    ```
13. 未达标处理：效果不明显 → 重新分析根因 → 二次改进

### Phase 5：自动化报告（1h）
14. 周报自动生成：
    - 本周工单量 / 解决量 / 积压量
    - Top 5 新增问题
    - 改进任务进度
    - 满意度趋势
15. 月报：产品质量评分（故障率、返修率）+ 改进成果汇总
16. 异常告警：某类问题突然激增 → 立即通知产品/研发

## 关键坑点
| 坑 | 后果 | 解法 |
|----|------|------|
| 分类不一致 | 同一问题被标记为不同分类 | 分类指南 + 客服培训 + AI 辅助 |
| 数据量不够 | 统计不显著 | 积累到 100+ 工单再做深度分析 |
| 改进不落地 | 分析了但没人执行 | 明确负责人 + 定期评审 + OKR 关联 |
| 验证周期长 | 硬件改进要等下一批次 | 区分快速修复（固件）和长期改进（硬件） |
| 归因困难 | 问题减少是改进效果还是自然波动 | 对照组 + 统计显著性检验 |
| 信息孤岛 | 售后数据和研发数据不互通 | 统一数据平台 + 定期同步会 |

## 与 RideWind 的关系
- 当前状态：无售后数据积累，问题处理靠记忆和微信聊天记录
- 优先级：P5（有工单系统后自然延伸）
- 起步方案：Excel/飞书表格手动记录 → 自动化看板
- 依赖：#25 工单系统（数据源）+ #49 远程诊断（设备数据）
- 价值：即使只有 20 台设备，记录问题也能发现共性
