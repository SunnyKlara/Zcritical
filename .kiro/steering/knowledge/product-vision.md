# 产品愿景 — Critical T1

> 这是产品层面的核心文档。AI 必须理解这里的内容才能做出符合产品方向的设计决策。

## 一句话定位

**桌面级智能汽车风洞模型** — 一款面向车模收藏玩家和汽车爱好者的桌面互动摆件，通过风扇、LED 灯效、引擎声浪、雾化烟雾，让静态车模"活"起来。

## 产品形态

- **硬件**：桌面尺寸的风洞腔体，内置风扇 + LED 灯带 + 扬声器 + 雾化器 + LCD 小屏 + 旋转编码器
- **软件**：手机 App（Android/iOS）通过蓝牙/WiFi 连接硬件，提供全部控制功能
- **使用场景**：放在书桌/展示柜上，摆入车模后开启风洞效果，日常观赏 + 拍照/拍视频 + 朋友来了展示

## 品类背景

桌面风洞模型是一个新兴品类（2024 年开始出现），代表产品是 Windsible（Fun-Tech-Lab，Kickstarter 众筹）。品类特征：
- 目标用户：1:18 / 1:24 / 1:64 比例车模收藏者、Hot Wheels 玩家、汽车爱好者
- 核心卖点：让静态车模有"动态感"（风吹 + 烟雾流线 + 灯光 + 声音）
- 价格区间：$130-$430（按尺寸）
- 竞品功能：风速旋钮调节、LED 灯光、烟雾发生器、透明展示腔

## Critical T1 的差异化

相比 Windsible 等竞品，Critical T1 的核心差异是**智能化 + App 深度控制 + 沉浸式体验**：

| 维度 | 竞品（Windsible） | Critical T1 |
|------|-------------------|-------------|
| 控制方式 | 物理旋钮 | 旋钮 + 手机 App 双控 |
| 灯光 | 固定 LED | 14 预设 + 自定义 RGB + 动态灯效（波浪/呼吸/舞台） |
| 声音 | 无 | 引擎声浪模拟（可变转速） + WiFi 音频投射 |
| 显示 | 无/简单 | 1.14" LCD 菜单 + 自定义 Logo |
| 联动 | 无 | 选车 → 自动匹配风速/灯光/声浪/Logo |
| 升级 | 不可 | OTA 固件升级 + App 自动更新 |
| 连接 | 无 | BLE + WiFi 双通道 |

## 核心体验关键词

- **沉浸感** — 风 + 光 + 声 + 雾 四感联动，不是单一效果
- **可玩性** — 油门操控模式让用户"开"车模，不只是看
- **个性化** — 灯光颜色、声浪风格、Logo 都可以自定义
- **收藏展示** — 日常作为精致桌面摆件，来客时一键开启表演

## App 功能定位

这是一款**桌面风洞设备专属配套控制 App**。

核心功能：
1. **蓝牙快速连接** — 一键匹配设备，实时同步状态
2. **趣味油门操控** — 模拟真实操控手感，无级调节风速
3. **全自定义 RGB 灯光** — 自由搭配色彩、亮度、动态效果
4. **个性化声浪设置** — 切换/调整引擎音效风格
5. **车库模式**（开发中）— 选车 → 一键联动全部参数
6. **Logo 上传** — 自定义 LCD 显示内容
7. **音频投射**（开发中）— 手机音乐通过设备扬声器播放

## 用户画像

- **核心用户**：20-40 岁男性，车模收藏爱好者，对汽车文化有热情
- **次要用户**：科技数码爱好者，喜欢桌面摆件和氛围灯
- **使用频率**：日常作为摆件常亮，偶尔切换车模/调整效果，朋友来时全开展示

## 设计原则（AI 做设计决策时参考）

1. **体验优先于功能数量** — 一个效果做到极致，比十个半成品强
2. **操作要简单** — 核心操作不超过 2 步（打开 App → 一键开启）
3. **视觉要精致** — App UI 和硬件效果都要有"高级感"，不是廉价玩具
4. **联动是核心** — 单独的风/光/声都不够，联动起来才有沉浸感
5. **硬件限制要尊重** — ESP32 资源有限，不追求手机级别的复杂度

## 四大功能的产品逻辑

### 车库（Garage）— 为什么做？
让用户"选一辆车 = 选一套完整体验"。不需要手动调风速、调灯光、调声浪，选车后全部自动匹配。这是产品从"工具"变成"玩具"的关键一步。

### Colorize 灯光升级 — 为什么做？
灯光是视觉冲击力最强的部分，也是拍照/拍视频时最重要的元素。更多灯效 = 更多可玩性 = 用户愿意分享。

### 音频投射 — 为什么做？
让设备变成一个"有灵魂的音箱"。日常不开风洞时也能当蓝牙音箱用，增加使用频率。

### iOS 开发 — 为什么做？
扩大用户群。车模收藏者中 iPhone 用户比例不低。同时为后续多平台（macOS 桌面配置工具等）做技术储备。

## 你否决过的方案（避免新 AI 重新提出）

- ❌ 引擎声用 MP3 循环播放 — 效果像"开关音箱"，改用 4 层可变采样率合成
- ❌ BLE 作为主通信通道 — 带宽不够，已决策切换到 WiFi WebSocket
- ❌ 换 Riverpod 状态管理 — 迁移成本太高，保持 Provider + get_it
- ❌ 改 Android 包名 — 会破坏 MethodChannel 和构建
- ❌ 用 PSRAM 做 OTA 缓冲 — Flash 写入时 PSRAM 不可访问
